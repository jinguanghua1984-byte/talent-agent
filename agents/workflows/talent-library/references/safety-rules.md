# talent-library 安全规则

1. 删除必须二次确认。删除前展示候选人摘要、关联数据计数和影响范围，并要求用户输入 `确认删除候选人 <candidate_id>`。
2. 批量写入、批量更新和批量删除必须先 dry-run，展示预计新增、修改、删除、跳过、失败和待人工确认数量。
3. 来源数据只追加，不静默覆盖已有来源；同一平台来源冲突时必须展示差异。
4. 评分必须写事件或记录。综合评分通过 `TalentDB.update_overall_score()` 写入评分事件；JD 匹配评分通过 `TalentDB.save_match_score()` 写入匹配记录。
5. 平台抓取和详情补全必须遵守 `agents/workflows/platform-match/AGENT.md` 的限流、session 和熔断规则。
6. 身份匹配、平台候选人匹配或详情置信度不足时，不自动合并、不自动写入详情。
7. 写库失败时必须报告失败原因、已完成范围、未完成范围和可重试建议。
8. 不绕过 TalentDB 拼 SQL；如现有 API 不足，报告能力缺口并停止对应写入。
9. 旧 JSON 删除必须单独确认；SQLite 删除不隐式删除旧 JSON。
10. 用户未确认前，不执行高风险覆盖：来源覆盖、评分覆盖、详情覆盖、批量字段修改和硬删除。
