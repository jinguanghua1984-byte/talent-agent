# 公域搜索

## 触发入口

```
/public-search [JD文本 | JD ID | 搜索策略 | 空]
```

### 入口判断

根据用户输入类型，自动路由到对应路径：

| 用户输入 | 路径 | 后续流程 |
|---------|------|---------|
| JD 文本 / JD ID | JD 驱动 | 读取 JD → 提取画像 → AI 生成策略 → 用户审查 → 确认执行 |
| 搜索策略描述（如"用 LinkedIn 搜 AI 产品经理"） | 策略驱动 | AI 审查策略 → 优化建议 → 用户确认执行 |
| 空 / 模糊岗位描述 | 引导模式 | AI 引导补充信息 → 生成策略 |

### JD 驱动路径

用户输入是 JD 文本或 JD ID 时：

1. 读取 JD 内容（ID 从 `data/jds/<id>.json` 读取，文本直接使用）
2. 提取关键信息：职位名称、技能要求、行业、公司类型、工作年限、城市
3. 生成 1-3 种候选人画像，每种包含核心特征
4. 基于画像生成搜索策略（参见「策略生成」章节）
5. 展示策略供用户审查和调整
6. 用户确认后执行搜索

### 策略驱动路径

用户输入已包含明确的搜索思路时：

1. 解析用户的搜索策略描述
2. AI 审查并补充优化建议（如遗漏渠道、关键词变体）
3. 展示完整策略供用户确认或调整
4. 用户确认后执行搜索

## 引导模式

当用户输入为空或内容不足以直接生成策略时，进入引导模式。

### 引导输出

```
你想搜什么人？可以给我：
1. 一个 JD（文本或 ID）
2. 你的搜索思路（如"用 LinkedIn 搜 AI 产品经理"）
3. 目标岗位和核心要求，我帮你生成策略
```

### 引导流程

1. 用户回复后，判断输入是否足够生成策略
2. 如信息不足，继续追问缺失的关键维度（目标岗位、核心技能、行业/公司偏好等）
3. 信息充足后，生成策略并展示供用户确认
4. 用户确认后进入执行阶段

### 退出机制

用户在引导模式中输入以下任一内容时，终止流程：

- 「取消」
- 「算了」
- 空消息（无任何输入）

退出时回复：**「好的，随时可以再来。」** 然后结束当前 Skill 执行。

## 协作策略生成

策略是用户与 AI 的**协作产物**，不是 AI 单方面输出。AI 的角色是审查、补充、优化，用户始终有最终决定权。

### 协作原则

- 策略文件是可编辑的活文档，用户可在任意阶段修改
- AI 给出的是「建议」，不是「决定」
- 用户确认后才进入执行阶段

### 策略文件格式

策略保存为活文档，格式严格遵循 `references/strategy-template.md` 中定义的 Instance 策略文件格式。

存储路径：`data/search-strategies/instances/{date}-{slug}.md`

### JD 驱动路径

用户输入 JD 文本或 JD ID 时，按以下步骤生成策略：

**Step 1：读取 JD，提取关键信息**

- JD 文本直接使用；JD ID 从 `data/jds/<id>.json` 读取
- 提取维度：职位名称、技能要求、行业、公司类型、工作年限、城市

**Step 2：匹配策略模板（Template）**

从 `data/search-strategies/templates/` 读取已有模板，匹配逻辑：

1. 提取新 JD 的特征（职位、技能、行业、经验）
2. 与每个模板的「共同特征」计算相似度
3. 相似度超过阈值（0.6）的模板为候选
4. 多个候选时按相似度排序，展示给用户选择
5. 用户也可选择「不使用模板」从零生成

**Step 3：应用 Universal 规则**

从 `data/search-strategies/universal-rules.json` 读取所有 `status: "active"` 的规则，在生成策略时作为过滤条件。例如：如果规则指出"不可量化指标命中率 0%"，则在关键词矩阵中标注此类关键词为低优先级。

**Step 4：生成初版策略文件**

基于画像、模板推荐（如有）、Universal 规则过滤结果，生成初版策略文件。策略文件写入 `data/search-strategies/instances/{date}-{slug}.md`，格式遵循 `references/strategy-template.md`。

**Step 5：用户审查、补充、修改**

展示策略文件内容给用户，用户可：
- 补充 AI 遗漏的关键词或渠道
- 修改优先级和权重
- 删除不合适的策略元素
- 覆盖岗位类型（记录在元信息中）

**Step 6：用户确认，进入执行**

用户确认定稿后，策略文件锁定当前版本，进入执行阶段。

### 策略驱动路径

用户输入已包含明确搜索策略时：

**Step 1：解析用户搜索策略，结构化展示**

将用户的策略描述解析为结构化格式（关键词矩阵、渠道计划），按策略文件格式展示。

**Step 2：AI 审查，给出优化建议**

AI 审查策略的完整性，标注建议（非决定）：
- 是否遗漏高优渠道
- 关键词是否有变体空间
- 是否与 Universal 规则冲突

**Step 3：用户采纳或拒绝，定稿**

用户逐条决定是否采纳 AI 的建议，形成最终策略。

**Step 4：用户确认，进入执行**

确认后写入策略文件，进入执行阶段。

### 跨会话恢复

搜索策略可能跨越多个会话完成，需支持中断恢复。

**恢复入口**

新会话启动时，用户给出 JD ID 或策略文件路径，AI 读取策略文件。

**恢复判断**

检查策略文件中的「最后活跃时间」：
- 距当前超过 24 小时：提示「这是一次 X 天前的搜索，是否继续？」
- 用户确认继续后，展示当前策略状态和执行进度

**并发控制**

策略文件中记录「当前轮次」，新会话开始时：
1. 读取文件中的轮次号
2. 如果与预期一致，正常继续
3. 如果不一致（说明有其他会话修改过），提示冲突，建议用户选择：
   - 基于当前文件继续
   - 创建新策略文件

**退出机制**

用户在策略生成流程中输入以下任一内容时，终止流程：

- 「取消」
- 「算了」
- 空消息（无任何输入）

退出时回复：**「好的，随时可以再来。」** 然后结束当前 Skill 执行。

## 执行搜索

策略确认后，按关键词矩阵逐条执行搜索。执行前**必须**参考 `references/search-sources.md` 中的渠道搜索语法和信息提取要点。

### 执行逻辑

按关键词矩阵逐条执行，每条独立记录：

1. 调用搜索工具（`WebSearch` / `mcp__ddg-search__search` / `mcp__github__search_users`）
2. 提取页面内容（`mcp__jina-reader__jina_reader` / `mcp__ddg-search__fetch_content`），识别候选人信息
3. 记录：搜索量、候选人数、噪音数
4. 候选人去重，合并 sources
5. 通过 Token Tracker 获取本次操作的 token 消耗（参见下方「Token 消耗追踪」）

### Token 消耗追踪

**方案：基于 Claude Code OpenTelemetry 遥测**

Skill 通过读取 collector 输出文件获取 token 数据。接口定义：

```typescript
interface TokenRecord {
  timestamp: string;
  prompt_id: string;
  input_tokens: number;
  output_tokens: number;
  cache_read_tokens: number;
  cache_creation_tokens: number;
  cost_usd: number;
  model: string;
}
```

**归因方式：** 每个 AI turn 对应一个 `prompt_id`。Skill 在执行搜索前后记录当前 turn 的 prompt_id，从 collector 输出中按 prompt_id 聚合 token 消耗，归因到对应的策略元素。

**降级方案：** 如果 OpenTelemetry 未配置，归因表中 Token 列显示「未配置」，成本分析跳过。不使用代理指标。

**前置任务：** Token Tracker 需要 OTEL 环境配置和本地 Collector 部署（独立任务，由华哥配合实施）。当前版本仅预留接口，未配置时优雅降级。

### 信息提取

从搜索结果中提取候选人信息：

| 字段 | 必须 | 来源 |
|------|------|------|
| 姓名 | 是 | 页面内容 |
| 当前公司 | 否 | 页面内容 |
| 当前职位 | 否 | 页面内容 |
| 城市 | 否 | 页面内容 |
| 技能标签 | 否 | 推断（技术栈、专业领域） |
| 来源渠道 | 是 | 搜索渠道名称 |
| 来源 URL | 是 | 搜索结果 URL |

提取规则：
- 姓名是唯一必填字段，其他字段按可用性填写
- 无法确定的字段留空，不要猜测
- 记录发现时间作为 `found_at`

### 候选人写入

用户确认搜索结果后，写入 `data/candidates/`：

1. 为每个候选人创建临时 JSON 文件
2. 调用 `python scripts/data-manager.py candidate create <file>` 写入
3. 清理临时文件
4. 多渠道发现的同一候选人合并 sources 而非重复创建
5. 去重逻辑复用 `python scripts/data-manager.py candidate dedup` 现有命令

## 搜索反馈

每轮搜索完成后，输出三层反馈结构。反馈数据全部来源于执行阶段记录的搜索量、候选人数、噪音数和 Token 消耗。

### 层 1：归因表

每个策略元素（关键词 + 渠道组合）的效果和成本可衡量：

```
| 策略元素 | 搜索量 | 候选人 | 噪音 | 命中率 | Token消耗 | 成本/候选人 | 状态 |
|---------|--------|--------|------|--------|----------|------------|------|
| "豆包" PM LinkedIn | 8 | 2 | 6 | 25% | 3.2K | 1.6K | ✅ 保留 |
| "AI PM" Google | 23 | 3 | 20 | 13% | 15.1K | 5.0K | ⚠️ 成本偏高 |
| "百万DAU" Google | 15 | 0 | 15 | 0% | 12.3K | ∞ | ❌ 建议移除 |
```

**状态判定规则：**

- 候选人 >= 2 且命中率 >= 20%：✅ 保留
- 候选人 >= 1 且命中率 >= 10%：⚠️ 成本偏高（Token 消耗高于平均值 2 倍时标注）
- 候选人 = 0：❌ 建议移除

**Token 消耗列规则：**

- Token Tracker 已配置时：显示实际消耗（如 `3.2K`）
- Token Tracker 未配置时：显示「未配置」

**成本/候选人列规则：**

- 候选人 > 0 时：Token 消耗 / 候选人数（如 `1.6K`）
- 候选人 = 0 时：显示 `∞`

### 层 2：覆盖缺口

对照 JD 要求（或策略文件中定义的目标画像），分析匹配情况：

```
| JD 要求 | 已匹配 | 缺口 |
|---------|--------|------|
| 5-10年产品经验 | ✅ 5人 | — |
| C端APP独立负责 | ⚠️ 2人 | 不确定是否独立负责 |
| 百万DAU+ | ❌ 0人 | 公开渠道难以验证 |
```

**已匹配列规则：**

- 匹配人数 >= 需求人数：✅ N人
- 匹配人数 > 0 但不足：⚠️ N人
- 匹配人数 = 0：❌ 0人

**缺口列规则：**

- 无缺口：显示 `—`
- 有缺口：简要描述原因

### 层 3：AI 反思 + 成本分析 + 优化建议

AI 从归因表和覆盖缺口数据中自动分析，给出建议但不直接修改策略。用户自行决定是否采纳。

**输出格式：**

```
🤖 本轮反思
• 渠道：LinkedIn(25%) > Google(13%)，产品经理岗位 LinkedIn 更有效
• 关键词：带公司名(豆包)比泛词(AI产品经理)更精准
• 缺口：百万DAU 在公开渠道不可验证

💰 成本分析
• 本轮总消耗：30.6K tokens ($0.09)
• 最高效："豆包" LinkedIn（1.6K/人）
• 最浪费："百万DAU" Google（12.3K，0产出）

💡 优化建议
A. 关键词：去掉"百万DAU"，加"ChatGPT Notion AI 产品经理"
B. 渠道：提高 LinkedIn 优先级
C. 画像：收窄到大厂AI部门PM

你的想法？
```

**AI 反思规则：**

- 不给结论，给选项。每个建议用字母标号，方便用户选择采纳
- 基于数据说话（命中率、Token 消耗），不凭直觉
- 成本分析在 Token Tracker 未配置时跳过「💰 成本分析」区块

## 迭代循环

搜索是一个迭代过程。每轮反馈后，用户选择下一步操作。

### 用户选择

| 用户反馈 | 动作 |
|---------|------|
| 采纳某些建议（如"采纳 A、C"） | 更新策略文件 → 再搜一轮 |
| 提出自己的优化（如"加上字节跳动的 PM"） | AI 审查优化合理性 → 更新策略 → 再搜一轮 |
| 满意当前结果（如"够了"） | 结束搜索 → 进入 screen/report 阶段 |
| 放弃（如"找不到合适的人"） | 在策略文件中记录放弃原因和轮次，不删除文件 |

### 每轮迭代操作

用户选择继续迭代时（采纳建议或自提优化），执行以下操作：

1. **更新执行记录**：在策略文件的「执行记录」中追加新一轮记录（轮次号、关键词变更、搜索结果摘要）
2. **更新关键词矩阵**：根据采纳的建议或用户的优化，更新关键词矩阵的「状态」和「备注」列（移除 ❌ 的、新增 ✅ 的）
3. **重新计算累计归因**：合并历史轮次的搜索数据，生成累计归因表（不是单轮归因）
4. **重新分析覆盖缺口**：基于累计候选人数据重新评估覆盖情况
5. **更新最后活跃时间**：刷新策略文件中的时间戳
6. **Git commit**：提交策略文件变更，commit message 格式 `search({jd-id}): 第N轮迭代 - {变更摘要}`

### 放弃记录

用户选择放弃时，在策略文件末尾追加放弃记录，**不删除策略文件**：

```markdown
## 搜索结果：放弃
- 放弃轮次：{N}
- 放弃原因：{用户描述}
- 沉淀经验：{AI 从本轮搜索中学到的通用经验}
```

**沉淀经验生成规则：**

- AI 从本轮搜索的归因数据中提取可复用的经验（如"某类关键词在公开渠道命中率极低"）
- 经验描述为具体、可操作的模式，不是泛泛的总结
- 放弃的 Instance 不参与策略模板（Template）的统计数据计算
- 沉淀经验可被 AI 提取用于 Universal 规则（需用户确认后写入 `data/search-strategies/universal-rules.json`）

## 策略沉淀

搜索经验通过三级结构逐步提炼：Instance（单次） → 策略模板（同类共性） → Universal 规则（跨模板共性）。每一级提升都由 AI 建议触发，用户决定是否采纳。

### 核心概念

**「同类搜索」的定义：** 两次搜索的 JD 全文相似度超过阈值（初始阈值：0.6，可调）。相似度计算方式：提取 JD 的职位名称、技能要求、行业、经验要求等字段，计算加权相似度。

**策略模板不是「岗位模板」：** 模板捕获的是近似搜索策略的共性特征，不局限于同一岗位。两个不同岗位如果搜索策略高度相似，可以归入同一模板。

### Instance → 策略模板

**触发条件：** 当 N 个 Instance 的 JD 两两相似度均超过阈值（初始 N=3，可调），AI 提议创建模板。已有模板下积累 N+ 次新 Instance 时，AI 建议更新模板统计数据。

**AI 提议格式：**

```
发现 3 次近似搜索的共性：
• LinkedIn 命中率稳定在 20%+
• GitHub 命中率接近 0%
• 带具体产品名的关键词效果最好

共同特征：C端产品经理、AI行业、上海

是否创建搜索策略模板？
```

**模板创建流程：**

1. AI 从 N 个相似 Instance 中提取共性（共同特征、推荐渠道、关键词模式、排除项）
2. 展示共性摘要，等待用户确认
3. 用户确认后，按 `references/strategy-template.md` 中的 Template 格式创建文件
4. 存储路径：`data/search-strategies/templates/{template-id}.md`
5. 关联 Instance 的元信息中更新「来源模板」字段

**模板更新流程：**

1. 新 Instance 完成搜索后，检查是否匹配已有模板
2. 如果匹配，AI 建议将该 Instance 的数据纳入模板统计
3. 更新模板的统计数据（关联 Instance 数、平均命中率、平均成本/候选人）
4. 如果新 Instance 的数据与模板推荐存在显著偏差（如某渠道命中率从 20% 降到 5%），AI 标注差异并建议用户审查

### 新 JD 匹配策略模板

匹配逻辑与「同类判定」是同一套：

1. 提取新 JD 的特征（职位、技能、行业、经验）
2. 与每个策略模板的「共同特征」计算相似度
3. 相似度超过阈值（0.6）的模板为候选
4. 多个候选模板时，按相似度排序，展示给用户选择
5. 用户也可选择「不使用模板」从零生成

### 策略模板 → Universal 规则

**触发条件：** 同一规律出现在 3+ 个策略模板中。

**AI 提议格式：**

```
发现跨模板共性规律：
• 规律：具体公司名/产品名作为关键词比泛词命中率更高
• 出现在 3 个模板中：template-001, template-003, template-007
• 置信度：高

是否添加为 Universal 规则？
```

**Universal 规则存储：** `data/search-strategies/universal-rules.json`

```json
{
  "rules": [
    {
      "id": "universal-001",
      "rule": "具体公司名/产品名作为关键词比泛词命中率更高",
      "based_on": ["template-001", "template-003"],
      "confidence": "high",
      "created_at": "2026-04-10",
      "status": "active"
    }
  ],
  "metadata": {
    "created_at": "2026-04-10",
    "last_updated": "2026-04-10"
  }
}
```

**规则生命周期：**

- `status` 枚举：`active` / `deprecated`
- AI 在每次搜索完成后检查 Universal 规则是否与最新 Instance 数据一致
- 连续 5 次 Instance 违反某规则 → AI 建议标记为 `deprecated`，用户确认后生效
- `deprecated` 规则保留在文件中（不删除），避免重复发现

### 沉淀全景

```
Instance（单次策略活文档）
  ↓ N 次相似 Instance（JD 全文相似度 > 0.6）
策略模板（共同特征 + 推荐策略 + 统计）
  ↓ 同一规律出现在 3+ 个模板
Universal 规则（跨模板共性）
```

**反向应用：**

```
新 JD → 匹配策略模板 → 获取推荐策略 → 应用 Universal 规则 → 生成 Instance → 用户协作 → 执行 → 迭代 → 沉淀
```

### 用户可控原则

- AI 建议提升（Instance → 策略模板 → Universal），用户决定是否采纳
- 用户可以手动创建/编辑/合并/删除策略模板
- 用户可以手动添加/删除/废弃 Universal 规则
- 相似度阈值用户可调

### 数据排除规则

- 放弃的 Instance 不参与策略模板的统计数据计算
- 放弃 Instance 的「沉淀经验」部分可被 AI 提取用于 Universal 规则（需用户确认）

## 岗位感知

### 岗位类型判断

从 JD 职位名称中提取关键词，匹配 `references/search-sources.md` 中的「岗位类型渠道推荐」表。

**判断流程：**

1. 从 JD 的职位名称字段提取关键词
2. 匹配岗位类型（技术开发 / 产品经理 / 设计师 / 管理-高管 / 学术-研究）
3. 模糊情况默认匹配最接近的类型
4. 用户可手动覆盖，覆盖记录在策略文件的元信息中（`岗位类型覆盖` 字段）

**岗位类型与渠道推荐：**

| 岗位类型 | 首选渠道 | 排除渠道 |
|---------|---------|---------|
| 技术开发 | GitHub, Google | Scholar |
| 产品经理 | LinkedIn, Google | GitHub, Scholar |
| 设计师 | Dribbble, Behance | GitHub, Scholar |
| 管理/高管 | LinkedIn, 行业媒体 | GitHub |
| 学术/研究 | Scholar, 知网 | — |

岗位感知影响策略生成的渠道选择。仅作推荐，用户可随时忽略。
