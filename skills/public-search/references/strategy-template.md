# 搜索策略文件格式参考

本文档定义 public-search Skill 中所有策略文件的格式规范。AI 在创建或更新策略文件时，必须遵循这些格式。

---

## 1. Instance 策略文件（活文档）

Instance 策略文件是每个 JD 的搜索策略记录，随搜索轮次不断更新。文件名建议使用 JD 标题的简短标识。

### 完整格式

```markdown
# 搜索策略：{JD标题}

## 元信息
- JD ID: {id}
- 创建时间: {timestamp}
- 当前轮次: {N}
- 来源模板: {template-id 或 "无"}
- 最后活跃时间: {timestamp}
- 岗位类型覆盖: {原始推荐 或 "用户覆盖为: X"}

## 目标画像
（1-3 种候选人画像，每种包含核心特征）

## 搜索策略
### 关键词矩阵
| # | 关键词 | 渠道 | 优先级 | 状态 | 备注 |
|---|--------|------|--------|------|------|
| 1 | "AI Agent" "产品经理" | Google | 高 | active | |
| 2 | "百万DAU" "产品经理" | Google | 高 | removed | 轮次2: 命中率0% |

### 渠道计划
| 渠道 | 搜索条数 | 权重 |
|------|---------|------|
| LinkedIn | 3 | 高 |

## 执行记录
### 轮次 1 ({timestamp})
#### 归因表
| 策略元素 | 搜索量 | 候选人 | 噪音 | 命中率 | Token消耗 | 成本/候选人 | 状态 |
|---------|--------|--------|------|--------|----------|------------|------|

#### 覆盖缺口
| JD 要求 | 已匹配 | 缺口 |
|---------|--------|------|

#### AI 反思
（本轮学到了什么、成本分析、优化建议）

### 累计归因
| 策略元素 | 累计搜索量 | 累计候选人 | 累计命中率 | 累计Token | 平均成本/人 | 状态 |
|---------|-----------|-----------|-----------|----------|------------|------|

### 轮次 2 ({timestamp})
...
```

### 字段说明

#### 元信息

| 字段 | 类型 | 说明 |
|------|------|------|
| JD ID | string | JD 的唯一标识符，用于关联和去重 |
| 创建时间 | timestamp | 首次创建策略文件的时间 |
| 当前轮次 | integer | 已执行的搜索轮次总数，每完成一轮 +1 |
| 来源模板 | string | 如果 JD 匹配了某个 Template，记录其 ID；否则为 "无" |
| 最后活跃时间 | timestamp | 每次更新策略时同步更新，用于跨会话恢复判断 |
| 岗位类型覆盖 | string | 初始为系统推荐的岗位类型；用户可手动覆盖，格式为 "用户覆盖为: X" |

#### 目标画像

- 1-3 种候选人画像
- 每种画像包含核心特征（如技术栈、行业经验、产品类型等）
- 画像基于 JD 要求推导，帮助聚焦搜索方向

#### 关键词矩阵

| 列 | 说明 |
|----|------|
| # | 序号，方便引用 |
| 关键词 | 搜索关键词组合，使用引号包裹精确匹配词 |
| 渠道 | 搜索渠道（Google / LinkedIn / GitHub 等） |
| 优先级 | 高 / 中 / 低，决定搜索执行顺序 |
| 状态 | `active`（当前轮次使用）/ `removed`（已移除）/ `modified`（已修改） |
| 备注 | 状态变更原因或其他补充信息 |

#### 渠道计划

| 列 | 说明 |
|----|------|
| 渠道 | 搜索渠道名称 |
| 搜索条数 | 计划在该渠道执行的搜索次数 |
| 权重 | 高 / 中 / 低，反映渠道对本次 JD 的重要程度 |

#### 执行记录 - 归因表

| 列 | 说明 |
|----|------|
| 策略元素 | 关键词或渠道名称，对应关键词矩阵中的条目 |
| 搜索量 | 该策略元素触发的搜索次数 |
| 候选人 | 找到的候选人数量 |
| 噪音 | 不相关的结果数量 |
| 命中率 | 候选人 / 搜索量，百分比 |
| Token消耗 | 该策略元素消耗的 Token 数量；为 0 时显示「未配置」 |
| 成本/候选人 | Token消耗 / 候选人，衡量效率 |
| 状态 | `active` / `removed` / `modified` |

#### 执行记录 - 覆盖缺口

| 列 | 说明 |
|----|------|
| JD 要求 | JD 中的具体要求 |
| 已匹配 | 已找到匹配该要求的候选人数量 |
| 缺口 | 尚未覆盖的缺口描述 |

#### 执行记录 - AI 反思

- 本轮学到了什么（有效/无效策略）
- 成本分析（Token 使用效率）
- 优化建议（给用户选项，不给结论）

#### 累计归因

- 多轮搜索后汇总每个策略元素的累计效果
- 位于每轮记录之后，汇总所有已执行轮次的数据
| 列 | 说明 |
|----|------|
| 策略元素 | 关键词或渠道名称 |
| 累计搜索量 | 所有轮次中该元素的搜索总量 |
| 累计候选人 | 所有轮次中该元素找到的候选人总数 |
| 累计命中率 | 累计候选人 / 累计搜索量 |
| 累计Token | 所有轮次中该元素的 Token 消耗总和 |
| 平均成本/人 | 累计Token / 累计候选人 |
| 状态 | `active` / `removed` / `modified` |

## 2. Template 策略文件

Template 从多个相似 Instance 中提取共性，用于新 JD 的快速策略初始化。文件名使用 template-id。

### 完整格式

```markdown
# 搜索策略模板：{template-id}

## 共同特征
（从相关 Instance 的 JD 中提取的共性，用于新 JD 匹配判断）
- 职位类型：产品经理
- 行业：AI
- 关键词：C端、移动端、Agent
- ...

## 推荐渠道
1. LinkedIn（权重：高，命中率 22%）
2. Google（权重：高，命中率 13%）

## 推荐关键词模式
- 具体产品名 > 泛词
- 带 AI 公司名 > 不带

## 排除渠道
- GitHub（命中率 < 2%）

## 排除关键词模式
- 不可量化指标（百万DAU等）命中率 0%

## 统计数据
- 关联 Instance 数：3
- 平均命中率：18%
- 平均成本/候选人：4.2K tokens
- 最后更新：YYYY-MM-DD
```

### 字段说明

| 字段 | 说明 |
|------|------|
| 共同特征 | 从关联 Instance 的 JD 中提取的共性特征，包括职位类型、行业、关键词等，用于新 JD 匹配判断 |
| 推荐渠道 | 按优先级排序的推荐搜索渠道，附带权重和历史命中率 |
| 推荐关键词模式 | 经验证有效的关键词构造策略，使用 `>` 表示优先级关系 |
| 排除渠道 | 命中率低于阈值的渠道，不推荐使用 |
| 排除关键词模式 | 经验证无效的关键词构造策略 |
| 统计数据 | 基于关联 Instance 的汇总统计，仅统计非放弃的 Instance |

## 3. 放弃记录

当用户决定放弃某个 JD 的搜索时，在 Instance 策略文件末尾追加放弃记录。

### 格式

```markdown
## 搜索结果：放弃
- 放弃轮次：{N}
- 放弃原因：{用户描述}
- 沉淀经验：{AI 从本轮搜索中学到的通用经验}
```

### 字段说明

| 字段 | 说明 |
|------|------|
| 放弃轮次 | 放弃时的当前搜索轮次 |
| 放弃原因 | 用户描述的放弃原因 |
| 沉淀经验 | AI 从本轮搜索中提取的通用经验，可用于 Universal 规则 |

### 规则

- 放弃的 Instance **不参与** Template 策略文件的统计数据计算
- 「沉淀经验」部分可被 AI 提取用于 Universal 规则
- 放弃记录追加在策略文件末尾，不删除已有的执行记录

## 4. 文件存储位置

| 文件类型 | 存储路径 |
|---------|---------|
| Instance 策略文件 | `data/talent/strategies/instances/{jd-id}.md` |
| Template 策略文件 | `data/talent/strategies/templates/{template-id}.md` |
