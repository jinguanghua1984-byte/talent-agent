# Career-Ops 项目调研分析报告

> 调研日期: 2026-04-28
> 项目地址: https://github.com/santifer/career-ops
> 项目性质: AI 驱动的求职命令中心（Prompt 工程 + Agent 工作流）
> 调研目标: 评估其匹配算法、架构设计对「简历与岗位JD匹配度评价系统」的可借鉴性

## 一、项目概述

### 1.1 项目定位

Career-Ops 是一个 **基于 LLM Agent 的求职全流程管理系统**，将 Claude Code / Gemini CLI / OpenCode 变成一个完整的求职 pipeline。它**不是**传统的 NLP 相似度匹配系统，而是一个 **Prompt 驱动的智能工作流**，用 LLM 推理能力作为核心匹配引擎。

**关键定位差异：**

| 维度 | Career-Ops | 我们的匹配度评价系统 |
|------|-----------|---------------------|
| 核心目标 | 个人求职全流程管理 | 简历与岗位的匹配度评分 |
| 匹配方式 | LLM 语义推理（非确定性） | 需要定量评分 + LLM 深度分析 |
| 输入 | 手写 Markdown CV + JD 文本/URL | PDF/DOCX 简历 + 结构化 JD |
| 输出 | 评分(1-5) + 完整评估报告 + 定制简历 | 匹配度分数 + 差距分析 |
| 用户 | 求职者（管理自己的求职） | HR/招聘方（评价候选人匹配度） |
| 规模 | 单用户 | 多候选人批量匹配 |
| 技术路线 | 纯 LLM Prompt 驱动 | LLM + 算法混合 |

### 1.2 技术栈

| 层面 | 技术选型 |
|------|---------|
| Agent 运行时 | Claude Code / Gemini CLI / OpenCode |
| 脚本语言 | Node.js (ESM `.mjs`) + Bash |
| Dashboard | Go + Bubble Tea + Lipgloss (TUI) |
| 依赖 | `@google/generative-ai`, `dotenv`, `js-yaml`, `playwright` |
| 数据格式 | Markdown (CV, 报告), YAML (配置), TSV (批量状态), HTML (简历模板) |
| PDF 生成 | Playwright 无头 Chromium 渲染 HTML 模板 |
| 岗位扫描 | 直接 HTTP 请求 ATS API (Greenhouse/Ashby/Lever) |

## 三、匹配算法详解（重点）

### 3.1 核心特点：纯 LLM 推理匹配

**Career-Ops 没有任何传统 NLP 匹配算法。** 没有 TF-IDF、BM25、余弦相似度、词向量。所有匹配完全依赖 LLM 的语义推理能力。

### 3.2 三步评分流程

#### Step 1: Archetype（原型）检测

每个 JD 被分类为 6 种原型之一（或混合两种）：

| 原型 | 关键信号词 |
|------|-----------|
| AI Platform / LLMOps | "observability", "evals", "pipelines", "monitoring" |
| Agentic / Automation | "agent", "HITL", "orchestration", "multi-agent" |
| Technical AI PM | "PRD", "roadmap", "discovery", "stakeholder" |
| AI Solutions Architect | "architecture", "enterprise", "integration" |
| AI Forward Deployed | "client-facing", "deploy", "prototype" |
| AI Transformation | "change management", "adoption", "enablement" |

**作用：** 同一份 CV 根据不同原型会采用不同的展示策略，这是自适应匹配的核心。

#### Step 2: 7-Block 评估 (A-G)

| 模块 | 内容 | 评分方式 |
|------|------|---------|
| **A -- 角色摘要** | 原型、领域、职能、级别、远程、团队规模 | 分类（无分数） |
| **B -- CV 匹配** | JD 要求映射到 CV 具体行 + 差距及缓解策略 | **核心匹配分析** |
| **C -- 级别策略** | JD 级别 vs 候选人自然级别，"sell senior" 计划 | 定性分析 |
| **D -- 薪资与需求** | WebSearch 薪资数据，市场需求 | 1-5 分 |
| **E -- 个性化计划** | Top 5 CV 修改 + Top 5 LinkedIn 修改 | 定性建议 |
| **F -- 面试计划** | 6-10 个 STAR+R 故事映射到 JD 要求 | 定性建议 |
| **G -- 招聘真实性** | 3 级评估（高置信 / 谨慎推进 / 可疑） | 定性评估 |

**Block B 是匹配的核心：** LLM 将 JD 的每一条要求与 CV 的具体行做精确映射，标注匹配/差距，并给出缓解策略。这不是简单的关键词匹配，而是深度的语义理解。

#### Step 3: 全局评分 (1-5)

5 个维度的加权评分：

| 维度 | 权重 | 数据来源 |
|------|------|---------|
| CV 匹配度 | 主要 | Block B 分析 |
| 北极星对齐 | 主要 | 用户目标原型（`_profile.md`） |
| 薪资 | 标准 | Block D 研究 |
| 文化信号 | 标准 | 公司文化、成长性、稳定性 |
| 红旗 | 负向调整 | 阻塞项、警告 |

**分数解读：**
- 4.5+ → 强匹配，建议立即申请
- 4.0-4.4 → 良好匹配，值得申请
- 3.5-3.9 → 尚可但不理想
- < 3.5 → 建议不申请

### 3.3 与 Resume-Matcher 的算法对比

| 维度 | Career-Ops | Resume-Matcher |
|------|-----------|---------------|
| 匹配方式 | LLM 语义推理 | 关键词命中率 |
| 确定性 | 非确定性（同一 JD 两次评分可能不同） | 确定性（同一输入同一结果） |
| 理解深度 | 深（语义理解、上下文推理） | 浅（字符串包含检查） |
| 评分维度 | 5 维加权 | 单维度（关键词命中率） |
| 可扩展性 | 差（每评一次消耗大量 token） | 好（算法轻量） |
| 可解释性 | 高（详细评估报告） | 低（只有百分比） |
| 验证难度 | 高（无基准数据集） | 低（可人工验证关键词） |

## 四、关键模块实现细节

### 4.1 简历处理（无传统解析）

Career-Ops **不解析简历**。用户手动维护 `cv.md`（Markdown 格式），LLM 直接读取原始文本进行推理。

```markdown
# cv.md 示例结构
## Summary
资深 AI 工程师，5 年 LLM 应用开发经验...

## Experience
### Company A | Senior AI Engineer | 2022-2025
- 设计并部署了 fraud detection ML pipeline...
- 构建了 multi-agent orchestration 系统...

## Skills
Python, PyTorch, LangChain, AWS...
```

**配套文件：**
- `article-digest.md` — 作品集精简证明点（含量化指标）
- `config/profile.yml` — 候选人画像（目标角色、薪资期望、地点偏好）
- `cv-sync-check.mjs` — CV 与配置一致性校验脚本

### 4.2 JD 提取（3 级策略）

| 级别 | 方法 | 适用场景 |
|------|------|---------|
| 1（首选） | Playwright 渲染 SPA | Ashby, Greenhouse, Lever, Workday |
| 2（备选） | WebFetch 静态抓取 | ZipRecruiter, 公司官网 |
| 3（兜底） | WebSearch 搜索 | 以上都失败时 |

**招聘门户扫描器 (`scan.mjs`)** — 零 token 成本：
- 直接 HTTP 请求 ATS API（Greenhouse/Ashby/Lever）
- 结构化 JSON 解析（标题、URL、地点）
- 无 LLM 参与，纯 API 调用
- 标题关键词过滤 + 去重

### 4.3 ATS 优化 PDF 生成

**实现：** `generate-pdf.mjs` + `templates/cv-template.html`

关键特性：
1. **关键词注入** — 用 JD 词汇重新表述已有经验（不编造技能）
2. **ATS Unicode 规范化** — 确保特殊字符不被 ATS 系统误读
3. **单列布局** — 兼容所有 ATS 解析器
4. **占位符系统** — HTML 模板使用 `{{placeholder}}` 替换机制

### 4.4 批量处理架构

**实现：** `batch/batch-runner.sh` + `batch/batch-prompt.md`

```
Conductor (batch-runner.sh)
├── 读取 jobs.tsv（待处理队列）
├── 检查锁文件（防重复处理）
├── 为每个 job 启动独立 Worker
│   └── Worker = claude -p --system-file batch-prompt.md
│       ├── 读取 cv.md + JD
│       ├── 执行 A-G 评估
│       ├── 生成报告 + PDF
│       └── 写入 tracker 追踪
└── 合并结果到 applications.md
```

**设计亮点：**
- 每个 Worker 完全自包含（prompt 即系统指令）
- TSV 状态管理，支持断点续跑
- 锁文件防止并发冲突
- Bash 并行控制

### 4.5 模式分析引擎

**实现：** `analyze-patterns.mjs`（约 400 行）

输出指标：
- 转化漏斗（评估 → 申请 → 面试 → Offer）
- 评分 vs 结果相关性
- 原型转化率
- Top 阻塞因素（地理限制、技术栈不匹配、级别、现场要求）
- 远程政策转化率
- 数据驱动的最低申请分数阈值

### 4.6 Pipeline 完整性工具链

| 脚本 | 功能 |
|------|------|
| `merge-tracker.mjs` | 合并批量 TSV 追加到 applications.md |
| `verify-pipeline.mjs` | 状态健康检查、链接验证、重复检测 |
| `dedup-tracker.mjs` | 去除重复条目 |
| `normalize-statuses.mjs` | 状态别名映射到标准值 |

## 五、可借鉴内容评估

### 5.1 可直接复用（即拿即用）

| 组件 | 文件 | 复用价值 | 适配工作量 |
|------|------|---------|-----------|
| ATS PDF 生成 | `generate-pdf.mjs` + `cv-template.html` | ★★★★★ | 低 — HTML 模板 + Playwright 渲染，框架无关 |
| 招聘门户扫描器 | `scan.mjs` | ★★★★☆ | 低 — 独立 Node.js 模块，依赖少 |
| 模式分析引擎 | `analyze-patterns.mjs` | ★★★★☆ | 中 — 需适配数据格式 |
| Pipeline 工具链 | `merge/verify/dedup/normalize` | ★★★☆☆ | 低 — 通用 Markdown 数据管道 |
| 批量处理模式 | `batch-runner.sh` + `batch-prompt.md` | ★★★☆☆ | 中 — Bash → Python 重写 |

### 5.2 值得借鉴的设计模式

| 模式 | 描述 | 借鉴价值 |
|------|------|---------|
| **双层架构** | 系统层(自动更新) + 用户层(永不覆盖) | ★★★★★ — 数据与逻辑分离，避免更新冲突 |
| **原型分类系统** | 6 种岗位原型 + 自适应展示策略 | ★★★★☆ — 可扩展到更多行业/岗位类型 |
| **7-Block 评估框架** | 结构化多维评估，非单一分数 | ★★★★★ — 比单一分数更全面、更有参考价值 |
| **零 token 扫描** | 直接调 ATS API 获取岗位列表 | ★★★★☆ — 大幅降低批量处理成本 |
| **关键词注入原则** | 用 JD 词汇重新表述已有经验，不编造 | ★★★☆☆ — 道德边界明确，可参考 |
| **渐进式 Onboarding** | 6 步收集用户上下文，逐步提升评估质量 | ★★★☆☆ — 用户体验设计参考 |
| **状态规范管理** | `states.yml` + 别名映射 + 标准化 | ★★★☆☆ — 简单有效的状态管理 |

### 5.3 需要改造才能使用

| 内容 | 改造方向 | 工作量 |
|------|---------|-------|
| 纯 LLM 匹配 | 补充确定性算法（BM25/嵌入向量）做基线 | 高 |
| 单用户设计 | 改为多候选人 → 单 JD 的批量匹配 | 高 |
| Markdown CV 输入 | 增加 PDF/DOCX 解析能力 | 中 |
| 无结构化 JD 解析 | 增加 JD 字段提取（薪资、要求、地点） | 中 |
| 评分非确定性 | 引入锚定机制，确保同输入同输出 | 高 |
| AI/ML 领域原型 | 扩展为通用行业原型分类系统 | 中 |
| CLI Agent 交互 | 改为 REST API / SDK 接口 | 高 |

### 5.4 不适用内容

| 内容 | 原因 |
|------|------|
| 面试计划生成 (Block F) | 超出匹配度评价范围 |
| 个性化简历修改建议 (Block E) | 面向求职者，非招聘方 |
| 招聘真实性检测 (Block G) | 非匹配度核心功能 |
| Go TUI Dashboard | 技术栈不匹配 |
| Gemini 评估脚本 | 简化版，质量不如 Claude |

## 六、与我们系统的整合建议

### 6.1 推荐借鉴的优先级

**P0（立即借鉴）：**
1. **7-Block 评估框架** — 将匹配评估从单一分数扩展为多维度结构化报告
2. **原型分类系统** — 建立岗位类型分类，实现自适应匹配策略
3. **双层架构** — 系统逻辑与用户数据分离

**P1（短期整合）：**
4. **ATS PDF 生成** — 直接复用生成匹配报告的 PDF 版本
5. **Pipeline 工具链** — 批量处理的状态管理、去重、健康检查
6. **零 token 扫描** — 接入 ATS API 降低成本

**P2（中期规划）：**
7. **模式分析** — 转化漏斗、评分相关性分析
8. **批量处理模式** — 并行评估 + 断点续跑
9. **关键词注入** — 简历优化建议中的用词调整

### 6.2 混合匹配架构建议

Career-Ops 证明了 LLM 深度语义匹配的价值，但也暴露了纯 LLM 方案的不足。建议采用**分层混合架构**：

```
┌─────────────────────────────────────────┐
│ Layer 3: LLM 深度分析                    │
│ (借鉴 Career-Ops 的 7-Block 评估)       │
│ - 语义理解、上下文推理                    │
│ - 差距分析、缓解策略                      │
│ - 生成详细评估报告                        │
├─────────────────────────────────────────┤
│ Layer 2: 嵌入向量语义匹配                │
│ (Career-Ops 缺失，需自建)               │
│ - 技能嵌入相似度                          │
│ - 经验领域语义匹配                        │
│ - 批量候选初筛                            │
├─────────────────────────────────────────┤
│ Layer 1: 确定性算法基线                   │
│ (Career-Ops 缺失，需自建)               │
│ - BM25 关键词匹配                        │
│ - 结构化字段比对（学历、年限、薪资）       │
│ - 硬性条件过滤                            │
└─────────────────────────────────────────┘
```

**执行策略：**
- Layer 1 做快速筛选（毫秒级，低成本）
- Layer 2 做语义排序（秒级，中等成本）
- Layer 3 对 Top N 候选做深度分析（分钟级，高成本）

### 6.3 评分体系融合建议

结合 Career-Ops 的多维评分 + 我们需要的确定性评分：

```python
class MatchScore:
    # 确定性分数（Layer 1-2）
    keyword_score: float       # 关键词命中率 [0-100]
    skill_similarity: float    # 技能嵌入相似度 [0-100]
    field_match: float         # 结构化字段匹配 [0-100]

    # LLM 评分（Layer 3）
    semantic_match: float      # 语义理解匹配 [1-5]
    experience_relevance: float # 经验相关度 [1-5]
    growth_potential: float    # 成长潜力 [1-5]

    # 综合分数
    overall_score: float       # 加权综合 [0-100]
    confidence: float          # 评分置信度 [0-1]
    tier: str                  # A/B/C/D 分级
```

## 七、总结

### Career-Ops 的核心价值

| 维度 | 评价 |
|------|------|
| **架构设计** | 优秀 — 双层分离、模块化、可维护 |
| **匹配深度** | 优秀 — LLM 语义推理远超关键词匹配 |
| **评估全面性** | 优秀 — 7-Block 框架覆盖求职全链路 |
| **工程成熟度** | 良好 — 740+ 真实评估，100+ 生成简历 |
| **可复用性** | 中等 — Prompt 工程难以直接移植，工具链可复用 |
| **生产就绪度** | 低 — 非确定性评分、单用户、无 API |

### 一句话结论

**Career-Ops 是一个优秀的 Prompt 工程范例，其评估框架设计思想（原型分类 + 多维评估 + 自适应策略）值得深度借鉴，但纯 LLM 匹配路线不适合直接用于生产级匹配系统。最佳策略是将其 LLM 深度分析层作为匹配系统的顶层补充，底层仍需确定性算法保证稳定性和可扩展性。**
