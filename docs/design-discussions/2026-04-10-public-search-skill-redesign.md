# public-search Skill 重设计讨论

> 2026-04-10 与华哥讨论，基于阿里云 AI Agent 产品经理 JD 的实际搜索过程发现的 Skill 设计问题。

## 背景

在为「阿里云 AI Agent 移动端产品经理」JD 执行全链路测试时，发现 public-search Skill 存在以下问题：

1. **策略是内部步骤**：Skill 内部生成搜索策略，用户参与度低，无法审查和优化
2. **缺少引导入口**：用户没有 JD 也没有策略时，直接报错或盲搜
3. **一次性执行**：搜索完就结束，没有执行后复盘和策略迭代机制
4. **渠道/关键词不够精准**：对产品经理岗位用了技术岗的搜索渠道（GitHub/Scholar）
5. **screen 评分维度不适配**：eval-criteria 偏技术岗，对产品经理等非技术岗不适用

---

## 一、三层交互模型

### 核心理念：策略驱动 + 人机协作 + 持续迭代

搜索策略不是 AI 内部的中间产物，而是用户和 AI 协作的"活文档"。

### 第一层：引导式启动（入口层）

用户触发 `/public-search` 时，根据输入判断入口：

| 用户输入 | 走哪条路 |
|---------|---------|
| 有明确搜索策略 | → 策略驱动 |
| 给了 JD（无策略） | → JD 驱动 |
| 什么都没给 | → 引导模式 |

**引导模式**：提示用户给出 JD 或搜索思路，不要盲搜。

### 第二层：协作生成策略（策略层）

两条路径汇合到同一个策略确认环节：

**JD 驱动路径：**
1. AI 读取 JD，生成初版搜索策略
2. 用户审查，提出补充/修改
3. AI 与用户策略合并，生成最终策略
4. 用户确认后执行

**策略驱动路径：**
1. 用户给出自己的搜索策略
2. AI 审查，给出优化建议（渠道、关键词、画像）
3. 用户采纳/拒绝建议
4. 定稿后执行

**关键原则：**
- 策略是可编辑的协作产物
- 用户始终有最终决定权
- AI 的角色是审查、补充、优化，不是替代

### 第三层：执行后迭代（迭代层）

搜索执行不是终点，而是新一轮迭代的起点：

```
执行搜索 → 展示结果 → 用户反馈 → 修改策略 → 再搜 → 多轮后提炼最佳策略
```

**迭代记录：**
- 每轮迭代更新策略文件（活文档）
- 记录每轮的有效/无效关键词和渠道
- 最终提炼为该类 JD 的推荐搜索模板

## 二、搜索反馈机制

### 核心难点

执行一轮搜索后，如何清晰传达「搜索策略与搜索结果之间的关系」，让用户能想到下一步优化方向，同时让 AI 也能自我反思。

### 解决方案：三层反馈结构

#### 层 1：策略→结果 归因表

每个策略元素（关键词+渠道组合）的效果可衡量：

| 策略元素 | 搜索量 | 候选人 | 噪音 | 命中率 | 评价 |
|---------|--------|--------|------|--------|------|
| "AI Agent 产品经理" Google | 23 | 3 | 20 | 13% | 偏低 |
| "豆包" "产品经理" LinkedIn | 8 | 2 | 6 | 25% | 尚可 |
| "百万DAU" Google | 15 | 0 | 15 | 0% | 无效 ❌ |

用户一眼看出哪些策略元素有效、哪些无效。

#### 层 2：JD 覆盖缺口分析

对照 JD 要求，看哪些维度找到匹配、哪些仍是缺口：

| JD 要求 | 已找到 | 缺口 |
|---------|--------|------|
| 5-10年产品经验 | ✅ 5人 | — |
| C端APP独立负责 | ⚠️ 2人 | 不确定是否独立负责 |
| 百万DAU+ | ❌ 0人 | 公开渠道难以验证 |

用户能发现：某些要求在公开渠道根本无法验证，需要换思路。

#### 层 3：AI 自我反思 + 优化建议

AI 从数据中自动分析，但给建议而非直接改策略：

```
💡 基于本轮结果，有3个优化方向：
A. 关键词调整：去掉"百万DAU"，加"ChatGPT Notion AI 产品经理"
B. 渠道权重：提高LinkedIn优先级，降低行业媒体
C. 画像收窄：专注搜"大厂AI部门产品经理"

你倾向哪个方向？或者有自己的想法？
```

## 三、策略知识管理架构

### 三层沉淀模型

#### 第 1 层：单次策略（Instance）

每次搜索的完整记录，包含所有轮次的策略+结果+反思。
- 忠实记录，不提炼
- 这是原始数据层

#### 第 2 层：岗位模板（Template）

同一类岗位搜索多次后，提炼出可复用的策略模板。
- 例：「AI产品经理」模板：LinkedIn优先、即刻辅助、避开GitHub、关键词用具体产品名
- 触发条件：3-5 次同类 Instance 的共性提炼
- 新 JD 匹配到某 Template → 自动推荐基础策略

#### 第 3 层：通用规则（Universal）

跨岗位类型的搜索经验法则。
- 例：「百万DAU 在公开渠道不可验证，改搜具体产品名」
- 例：「LinkedIn 对产品/管理岗命中率最高，对技术岗一般」
- 所有搜索自动应用，不需要用户感知

### 三层关系

```
Instance（单次搜索）
  ↓ 积累 3-5 次同类
Template（岗位模板）
  ↓ 多个 Template 交叉
Universal（通用规则）
```

### 反向应用（新搜索流程）

```
新 JD 进来
  → 匹配最相关的 Template → 获得基础策略
  → 应用 Universal 规则 → 过滤已知无效做法
  → 生成 Instance 策略 → 用户协作确认
  → 执行 → 迭代 → 沉淀回 Template 和 Universal
```

### 关键设计点

1. **用户可控**：AI 可以建议提升（Instance → Template → Universal），但用户决定是否采纳
2. **Template 匹配**：按职位关键词匹配最相关的 Template，而非精确匹配
3. **不会过度抽象**：Template 至少需要 3 次同类搜索才生成，避免以偏概全
4. **效果待验证**：具体效果需要执行一段时间后再评估

## 四、讨论中确认的决策

| 决策项 | 结论 |
|--------|------|
| 策略是否协作生成 | ✅ 是，用户和 AI 共同编辑 |
| 反馈结构是否三层 | ✅ 归因表 + 覆盖缺口 + AI反思 |
| 策略是否分层沉淀 | ✅ Instance → Template → Universal |
| 用户是否可控 | ✅ AI 建议但不替代，用户最终决定 |
| Template 生成门槛 | ✅ 至少 3 次同类搜索 |
