## 这份指南解决什么问题

人才库会持续增加候选人、详情、推荐记录和删除记录。如果你在多台电脑上工作，或者团队里多人都在维护人才库，就需要把这些变化同步到同一个云端位置。

云端同步的作用是：

- 把本机新增的人才数据安全上传到飞书 Drive。
- 把别人上传的新数据合并到你的本机人才库。
- 避免重复导入同一批候选人。
- 发现冲突时先停下来，不自动覆盖你的本地判断。
- 让删除记录也能同步，避免已删除的人在另一台电脑上又出现。

这份指南只讲“怎么用”和“怎么判断结果”，不要求你理解数据库结构。

## 先记住 3 条规则

### 1. 不要手动上传 `data/talent.db`

`data/talent.db` 是系统内部数据库文件。不要把它直接拖到飞书 Drive，也不要用网盘自动同步这个文件。

正确做法是让系统生成加密同步包，再上传到飞书 Drive。同步工具会自动完成这一步。

### 2. 每次开工先同步，收工再同步

最简单的习惯：

- 开始工作前：先同步一次，把云端最新数据拉下来。
- 完成导入、补详情、删除或整理后：再同步一次，把你的变化推上去。

### 3. 看到冲突就先停

如果系统提示 `blocked`、`conflict` 或“冲突”，不要反复重试。冲突表示两边对同一个人的信息有不同判断，需要人工确认。

## 云端同步是怎么工作的

可以把云端同步理解成“加密信封”：

1. 系统把本机人才库变化打包成一个同步包。
2. 系统用同步密钥把同步包加密。
3. 加密后的文件上传到飞书 Drive。
4. 另一台电脑从飞书 Drive 下载加密文件。
5. 系统用同一个同步密钥解密。
6. 系统先预览有没有冲突，确认安全后再合并进本机人才库。

飞书 Drive 上看到的不是原始人才库数据库，而是加密后的同步文件和索引文件。

## 第一次使用前需要谁来准备

第一次设置建议让技术同事或项目负责人操作一次。业务同学只需要确认三件事：

| 需要准备的东西 | 谁负责 | 说明 |
| --- | --- | --- |
| 飞书 Drive 同步文件夹 | 项目负责人或技术同事 | 用来存放加密同步包 |
| 同步密钥 | 技术同事 | 所有参与同步的电脑使用同一个密钥 |
| 每台电脑的同步配置 | 技术同事 | 配好后日常只需要执行同步 |

### 同步密钥要怎么保管

同步密钥相当于打开加密同步包的钥匙。

- 可以放在团队密码管理工具里。
- 不要发到群聊。
- 不要写进文档。
- 不要提交到 Git。
- 如果密钥丢了，旧同步包无法解密，只能重新初始化一套同步配置。

### 技术同事第一次会做什么

通常会做这些动作：

```bash
.venv/bin/python -m scripts.talent_cloud_sync keygen
```

生成同步密钥后，在每台电脑上配置飞书文件夹和密钥，再初始化云端目录：

```bash
.venv/bin/python -m scripts.talent_cloud_sync init-remote --provider feishu
```

初始化后，飞书 Drive 里会出现一组系统目录，例如 `bundle-index`、`bundles`、`_meta`。这些目录不要手动改名、移动或删除。

### 第一次同步和日常同步

第一次把一台电脑的数据放到飞书 Drive 时，使用 full bootstrap：

```bash
.venv/bin/python -m scripts.talent_cloud_sync push --provider feishu --mode full
```

其它电脑第一次使用时，先拉取这个全量 bootstrap：

```bash
.venv/bin/python -m scripts.talent_cloud_sync pull --provider feishu
```

日常同步使用增量：

```bash
.venv/bin/python -m scripts.talent_cloud_sync sync --provider feishu
```

`sync` 会先拉取远端未应用的 bundle，再上传本机增量。如果远端已有本机没拉取的 bundle，系统会拒绝 push，要求先 pull。

如果本机没有任何变化，增量 push 会返回 no-op，不上传空包。

## 日常同步怎么做

日常使用推荐直接让 agent 执行同步。

你可以这样说：

```text
帮我同步人才库
```

或者说得更明确：

```text
先从飞书拉取人才库最新数据，再把我本机的更新推送到飞书
```

如果需要自己运行命令，使用：

```bash
.venv/bin/python -m scripts.talent_cloud_sync sync --provider feishu
```

这个命令会做两件事：

1. 先从飞书 Drive 拉取别人上传的新同步包。
2. 如果没有冲突，再把你本机的新变化上传到飞书 Drive。

### 什么时候一定要同步

| 场景 | 应该怎么做 |
| --- | --- |
| 开始一天工作前 | 先同步 |
| 导入新候选人后 | 同步 |
| 批量补充候选人详情后 | 同步 |
| 删除候选人后 | 同步 |
| 另一位同事刚说他更新了人才库 | 先同步再继续操作 |
| 准备换电脑工作 | 旧电脑先同步，新电脑再同步 |

## 多人协作时怎么同步

多人同时维护人才库时，建议按下面方式协作。

### 普通日常更新

每个人开工前同步一次，收工前同步一次。

### 批量导入前

如果你准备导入一大批候选人，先同步：

```text
我准备批量导入候选人，先帮我同步人才库
```

导入完成后再同步：

```text
导入完成了，把本机人才库同步到飞书
```

### 两个人都在改同一批候选人

尽量约定顺序：

1. A 先完成修改并同步。
2. B 先同步 A 的更新。
3. B 再继续修改并同步。

这样可以减少冲突。

## 看到这些结果说明正常

同步完成后，通常会看到类似这些状态。

| 结果 | 含义 |
| --- | --- |
| `applied` | 从云端成功合并了几个同步包 |
| `uploaded` | 本机是否上传了新的同步包 |
| `skipped` | 有些同步包已经导入过，本次跳过 |
| `unchanged` | 本机数据没有变化，不需要重复上传 |
| `blocked` | 有冲突或校验问题，本次没有自动写入 |

正常情况不一定每次都有新上传。如果你刚同步过，下一次看到 `unchanged` 或 `skipped` 是正常的。

### 同步前检查

如果怀疑飞书授权、空间或配置有问题，可以让 agent 先检查：

```text
帮我检查人才库云同步是否可用
```

对应命令是：

```bash
.venv/bin/python -m scripts.talent_cloud_sync doctor --provider feishu
```

检查通过后再同步。

## 遇到冲突怎么办

冲突通常表示同一个候选人在两台电脑上都有变化，而且系统不能确定应该保留哪一边。

常见例子：

- 你这边记录候选人在上海，同事那边记录在北京。
- 你这边删除了候选人，同事那边刚补了详情。
- 两边都改了候选人的核心信息。

看到冲突后：

1. 不要继续反复同步。
2. 不要手动上传或覆盖 `data/talent.db`。
3. 把冲突结果发给项目负责人或技术同事。
4. 由负责人确认保留哪边信息，再执行确认式导入或人工修正。

如果你使用 agent，可以直接说：

```text
人才库云同步出现冲突，帮我只读查看冲突摘要，不要自动写库
```

## 常见问题

### 我能不能直接把人才库文件放进飞书同步盘？

不能。直接同步 `data/talent.db` 容易造成数据库损坏或数据互相覆盖。

### 为什么还要加密？

人才库里可能包含候选人姓名、公司、职位、联系方式和沟通记录。即使飞书 Drive 有权限控制，也应该只上传加密后的同步包。

### 我换电脑了怎么办？

让技术同事在新电脑上配置同一个飞书同步文件夹和同一个同步密钥，然后先执行一次同步。

### 我忘了同步就开始导入了怎么办？

先不要再做更多操作。让 agent 执行一次同步。如果出现冲突，按冲突流程处理。

### 飞书空间满了怎么办？

先停止同步，让管理员检查飞书 Drive 空间。不要手动删除 `bundles` 或 `bundle-index` 里的文件，避免破坏同步历史。

### 同步会不会把别人删除的人恢复回来？

正常不会。删除记录也会进入同步包，系统会用删除记录防止旧数据复活。

### 我应该多久同步一次？

如果只有你一个人用，每天开工和收工各一次就够。多人协作时，开始批量导入前、完成批量导入后、收到同事更新通知后都应该同步。

---

## 一句话版流程

开工前说：

```text
帮我同步人才库
```

完成导入、补详情或删除后再说：

```text
把本机人才库同步到飞书
```

如果出现冲突，说：

```text
只读查看人才库同步冲突，不要自动写库
```
