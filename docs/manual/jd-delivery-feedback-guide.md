# 推荐人选反馈使用说明

> 给业务、猎头顾问和招聘同学使用：看完推荐人选后，只在外联表的 `feedback_note` 列写一句自然语言反馈，系统会自动解析成算法闭环需要的结构化字段。

---

## 这份反馈用来做什么

推荐系统会先根据 JD 生成候选人列表和 outreach 表。你对候选人的反馈会用于回答这些问题：

- 哪些推荐人选值得继续推进？
- 哪些人选看起来分高，但实际不适合当前 JD？
- 不匹配是因为岗位画像理解偏了，还是评分卡、匹配证据或搜索方向需要调整？
- 下一轮推荐应该放宽、收紧，还是换候选人来源或关键词？

这份反馈只评估猎头顾问对推荐结果的主观判断，不追踪长期外联周期。因此它不是 CRM 跟进表，也不要求记录是否已联系、是否已推客户、是否面试或是否 offer。

---

## 你只需要填写一个字段

业务侧只填写 `feedback_note`。

旧版反馈列已经废弃，不需要再填写：

- `feedback_label`
- `feedback_stage`
- `reason_codes`
- `hunter_note`
- `contacted`
- `submitted_to_client`
- `interviewed`
- `offer`

系统会从 `feedback_note` 自动解析出内部字段：`feedback_label`、`feedback_stage`、`reason_codes`、`hunter_note` 和 `parse_confidence`。这些字段只给系统复盘和算法校准使用，不需要业务手动选择或回填。

---

## 外联表里哪些列不要改

表里会包含候选人基础信息、推荐排名和评分信息。用于反馈回收的关键列包括：

| 列 | 谁维护 | 说明 |
| --- | --- | --- |
| `candidate_id` | 系统 | 候选人唯一标识，不要修改。 |
| `rank` | 系统 | 推荐排名，必须是正整数，不要修改。 |
| `grade` | 系统 | 推荐等级，只能是 `A`、`B`、`C`、`淘汰`，不要修改。 |
| `score` | 系统 | 原始推荐分数，不要修改。 |
| `feedback_note` | 业务 | 只填写这一列。 |

如果没有看过某个候选人，可以把 `feedback_note` 留空。空反馈会被跳过，不会进入本轮反馈统计。

---

## 怎么写 `feedback_note`

每一行代表一个候选人。建议用一句话写清楚三件事：

- 总体判断：认可、待定或不认可。
- 具体理由：为什么合适或不合适。
- 下一步建议：优先联系、再确认，还是不建议推进。

好的反馈：

```text
方向匹配，有大模型平台经验，建议优先联系，重点确认是否负责过平台能力建设。
```

不够好的反馈：

```text
还行
```

如果不确定，也可以直接说明不确定点：

```text
公司和标题相关，但看不出是否真正负责核心模块，需要电话确认。
```

---

## 常见场景示例

| 场景 | 可以这样写 |
| --- | --- |
| 认可 | 方向匹配，有大模型平台经验，近期经历和 JD 匹配，建议优先联系。 |
| 待定 | 标题相关，但简历没有写清是否负责核心模块，需要电话确认。 |
| 不认可 | 关键词命中了，但实际做销售支持，不是我们要找的平台产品负责人。 |
| 资历不符 | 背景相关，但目前更像执行层产品经理，和负责人岗位有差距。 |
| 证据不足 | 公司和项目看起来相关，但推荐报告里的证据太浅，暂时不能判断。 |
| 状态不明 | 方向匹配，但近期状态不清楚，需要先确认是否看机会。 |
| 信息过期 | 候选人历史经历匹配，但最近两段经历缺失，建议先更新信息再判断。 |

## 系统会怎么处理反馈

回收外联表后，系统会读取每条非空 `feedback_note`。明确、常见的短语会先由规则解析；复杂或不确定的反馈会批量交给 AI 解析器生成内部结构化结果：

| 内部字段 | 含义 | 业务是否需要填写 |
| --- | --- | --- |
| `feedback_label` | 总体判断：`认可`、`待定`、`不认可`。 | 不需要 |
| `feedback_stage` | 问题所属阶段：画像、评分卡、匹配、报告或候选人状态。 | 不需要 |
| `reason_codes` | 算法复盘用的原因码。 | 不需要 |
| `hunter_note` | 给人工复盘看的中文摘要。 | 不需要 |
| `parse_confidence` | AI 对解析结果的置信度，范围 0 到 1。 | 不需要 |

高置信度且字段合法的反馈会进入 `feedback/delivery-feedback.json`，再生成：

- `feedback/feedback-summary.json`
- `feedback/calibration-suggestions.json`

这些结果用于计算 `accepted_at_10`、`accepted_at_30`、`bad_at_10`、原因分布和不同 grade 的认可率。

---

## 哪些反馈会进入人工复核队列

以下情况会进入 `feedback/parse-review-queue.json`，默认不会进入本轮校准统计：

- AI 解析置信度低于 `0.7`。
- AI 返回了不合法字段，系统把结果降级为待复核。
- 规则无法明确判断，且批量 AI 解析结果仍不够确定。
- 反馈过短、过泛，无法判断具体原因。
- 反馈里同时出现互相冲突的判断。
- AI 调用失败。

进入复核队列不代表反馈无效，只表示这条反馈需要人工确认后再进入算法闭环。

---

## 注意事项

- 写候选人和岗位相关的专业判断，不写无关个人评价。
- 不要在 `feedback_note` 里写内部敏感信息、客户隐私或不适合公开流转的内容。
- 不需要写英文原因码，也不需要按固定格式填写。
- 不要改 `candidate_id`、`rank`、`grade`、`score` 等系统生成列。
- 不要把触达进展写成反馈主内容；本轮只评估推荐人选是否适合当前 JD。

---

## 常见问题

### 必须每个候选人都填吗？

不必须。优先填写你看过、判断明确、准备推进或明确不推进的人。

### 需要写固定格式吗？

不需要。自然语言即可。建议包含“是否认可 + 为什么 + 下一步”。

### 还需要填旧版 8 个反馈列吗？

不需要。当前业务表只保留 `feedback_note` 作为反馈入口，结构化字段由 AI 自动解析生成。

### `待定` 会不会影响算法？

会被记录，但低置信度或需要复核的 `待定` 会先进入 `parse-review-queue.json`，不会默认进入本轮校准统计。明确写出“不确定的原因”会更有价值。

### 反馈会直接发给候选人或客户吗？

不会。`feedback_note` 用于内部复盘和下一轮推荐校准。

### 后续系统会自动改人才库或评分卡吗？

不会。反馈只生成校准建议，不直接写入 `data/talent.db`，也不会自动修改评分卡或自动发布到知识库。
