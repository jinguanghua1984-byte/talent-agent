# public-search Skill 完全重写设计规格

> 基于 2026-04-10 全链路测试讨论，将 public-search 从「单次搜索工具」改造为「策略驱动的协作搜索系统」。

**状态：** Draft
**日期：** 2026-04-10
**决策：** 完全重写（非渐进改造）、策略数据存储在 talent-agent 项目内

---

## 1. 问题与目标

### 现状问题

| # | 问题 | 影响 |
|---|------|------|
| P1 | 策略是 AI 内部步骤，用户无法参与 | 搜索方向偏离用户意图 |
| P2 | 无引导入口 | 没有 JD 也没有策略时盲搜 |
| P3 | 一次性执行，无复盘迭代 | 同样的错误反复犯 |
| P4 | 渠道/关键词不区分岗位类型 | 产品经理搜 GitHub 无意义 |
| P5 | 搜索经验不积累 | 每次 from scratch |
| P6 | 不考虑搜索成本 | Token 消耗无感知，浪费真金白银 |

### 设计目标

1. **策略先行**：没有确认的策略不执行搜索
2. **人机协作**：策略是用户和 AI 共同编辑的产物
3. **效果可衡量**：每个策略元素的命中率和 token 成本可归因
4. **经验可复用**：搜索策略三层沉淀（Instance → Template → Universal）
5. **岗位感知**：根据岗位类型自动调整渠道和评估维度
6. **成本感知**：每次搜索记录 token 消耗，支持成本效益分析

## 2. 触发与入口

### 触发方式

```
/public-search [JD文本 | JD ID | 搜索策略 | 空]
```

### 入口判断

| 用户输入 | 路径 | 后续 |
|---------|------|------|
| JD 文本 / JD ID | JD 驱动 | AI 生成策略 → 用户审查 → 确认 |
| 搜索策略描述（如"用LinkedIn搜AI产品经理"） | 策略驱动 | AI 审查 → 优化建议 → 用户确认 |
| 空 / 岗位描述 | 引导模式 | AI 引导补充信息 → 生成策略 |

### 引导模式输出

```
你想搜什么人？可以给我：
1. 一个 JD（文本或 ID）
2. 你的搜索思路
3. 目标岗位和核心要求，我帮你生成策略
```

## 3. 协作策略生成

### 3.1 JD 驱动路径

```
Step 1: 读取 JD → 提取关键信息（职位、技能、行业、年限、城市）
Step 2: 匹配 Template（如有相关岗位模板）→ 获取推荐策略
Step 3: 应用 Universal 规则 → 过滤已知无效做法
Step 4: 生成初版策略文件 → 展示给用户
Step 5: 用户审查、补充、修改 → 合并定稿
Step 6: 用户确认 → 进入执行
```

### 3.2 策略驱动路径

```
Step 1: 解析用户搜索策略 → 结构化展示
Step 2: AI 审查 → 给出优化建议（标注"建议"非"决定"）
Step 3: 用户采纳/拒绝 → 定稿
Step 4: 用户确认 → 进入执行
```

### 3.3 协作原则

- 策略是可编辑的协作产物，不是 AI 单方面输出
- 用户始终有最终决定权
- AI 的角色是审查、补充、优化，不是替代

### 3.4 策略文件格式

策略保存为「活文档」，搜索过程中持续更新：

```markdown
# 搜索策略：{JD标题}

## 元信息
- JD ID: {id}
- 创建时间: {timestamp}
- 当前轮次: {N}
- 来源模板: {template-id 或 "无"}

## 目标画像
（1-3 种候选人画像，每种包含核心特征）

## 搜索策略
### 关键词矩阵
| # | 关键词 | 渠道 | 优先级 | 状态 |
|---|--------|------|--------|------|
| 1 | "AI Agent" "产品经理" | Google | 高 | active |
| 2 | "百万DAU" "产品经理" | Google | 高 | removed(轮次2: 命中率0%) |

### 渠道计划
| 渠道 | 搜索条数 | 权重 |
|------|---------|------|
| LinkedIn | 3 | 高 |

## 执行记录
### 轮次 1 ({timestamp})
#### 归因表
| 策略元素 | 搜索量 | 候选人 | 噪音 | 命中率 | Token消耗 | 成本/候选人 | 状态 |
|---------|--------|--------|------|--------|----------|------------|------|
#### 覆盖缺口
| JD 要求 | 已匹配 | 缺口 |
|---------|--------|------|
#### AI 反思
（本轮学到了什么、成本分析、优化建议）

### 轮次 2 ({timestamp})
...
```

### 3.5 存储路径

```
data/search-strategies/
├── instances/              # 单次搜索策略
│   └── {date}-{slug}.md
├── templates/              # 岗位模板（3-5次后提炼）
│   └── {role-type}.md
└── universal-rules.json    # 通用规则（跨岗位共性）
```

## 4. 执行搜索

### 4.1 执行逻辑

按关键词矩阵逐条执行，每条独立记录：

```
1. 调用搜索工具（WebSearch / GitHub / jina-reader）
2. 提取页面内容，识别候选人信息
3. 记录：搜索量、候选人数、噪音数、token 消耗
4. 候选人去重（同名+公司判断），合并 sources
```

### 4.2 Token 消耗追踪

每次搜索操作记录 token 消耗：

| 操作 | 记录内容 |
|------|---------|
| 搜索工具调用 | 搜索 token（输入 query 大小） |
| 页面内容提取 | 提取 token（返回内容大小） |
| AI 分析判断 | 分析 token（上下文消耗） |

汇总到归因表的「Token消耗」和「成本/候选人」列。

### 4.3 候选人写入

用户确认搜索结果后，写入 `data/candidates/`：

1. 为每个候选人创建临时 JSON 文件
2. 调用 `data-manager.py candidate create` 写入
3. 清理临时文件
4. 多渠道发现的同一候选人合并 sources 而非重复创建

## 5. 搜索反馈

### 5.1 层 1：归因表

每个策略元素（关键词+渠道组合）的效果和成本可衡量：

```
| 策略元素 | 搜索量 | 候选人 | 噪音 | 命中率 | Token消耗 | 成本/候选人 | 状态 |
|---------|--------|--------|------|--------|----------|------------|------|
| "豆包" PM LinkedIn | 8 | 2 | 6 | 25% | ~3K | 1.5K | ✅ 保留 |
| "AI PM" Google | 23 | 3 | 20 | 13% | ~15K | 5K | ⚠️ 成本偏高 |
| "百万DAU" Google | 15 | 0 | 15 | 0% | ~12K | ∞ | ❌ 建议移除 |
```

### 5.2 层 2：覆盖缺口

对照 JD 要求，分析匹配情况：

```
| JD 要求 | 已匹配 | 缺口 |
|---------|--------|------|
| 5-10年产品经验 | ✅ 5人 | — |
| C端APP独立负责 | ⚠️ 2人 | 不确定是否独立负责 |
| 百万DAU+ | ❌ 0人 | 公开渠道难以验证 |
```

### 5.3 层 3：AI 反思 + 成本分析 + 优化建议

AI 从数据中自动分析，给建议但不直接改策略：

```
🤖 本轮反思
• 渠道：LinkedIn(25%) > Google(13%)，产品经理岗位 LinkedIn 更有效
• 关键词：带公司名(豆包)比泛词(AI产品经理)更精准
• 缺口：百万DAU 在公开渠道不可验证

💰 成本分析
• 本轮总消耗：~30K tokens
• 最高效："豆包" LinkedIn（1.5K/人）
• 最浪费："百万DAU" Google（12K，0产出）

💡 优化建议
A. 关键词：去掉"百万DAU"，加"ChatGPT Notion AI 产品经理"
B. 渠道：提高 LinkedIn 优先级
C. 画像：收窄到大厂AI部门PM

你的想法？
```

## 6. 迭代循环

### 用户选择

| 用户反馈 | 动作 |
|---------|------|
| 采纳某些建议 | 更新策略文件 → 再搜一轮 |
| 提出自己的优化 | AI 审查 → 更新策略 → 再搜一轮 |
| 满意当前结果 | 结束搜索 → 进入 screen/report |
| 放弃 | 记录失败原因 → 沉淀经验 |

### 每轮迭代操作

1. 更新策略文件中的「执行记录」（追加新一轮）
2. 更新关键词矩阵的「状态」列（active/removed/modified）
3. 重新计算累计归因表
4. 重新分析覆盖缺口

## 7. 策略沉淀

### 7.1 Instance → Template

**触发条件：** 同一岗位类型积累 3+ 次 Instance 搜索。

**AI 提议：**
```
💡 发现 3 次「AI产品经理」搜索的共性：
• LinkedIn 命中率稳定在 20%+
• GitHub 命中率接近 0%
• 带具体产品名的关键词效果最好

是否创建「AI产品经理」岗位模板？
```

**Template 文件格式：**
```markdown
# 岗位模板：{岗位类型}

## 适用条件
- 职位关键词包含：...
- 行业关键词包含：...

## 推荐渠道
1. LinkedIn（权重：高）
2. Google（权重：高）

## 推荐关键词模式
- 具体产品名 > 泛词
- 带 AI 公司名 > 不带

## 排除渠道
- GitHub（命中率 < 2%）

## 统计数据
- 基于搜索次数：N
- 平均命中率：X%
- 平均成本/候选人：YK tokens
- 最后更新：YYYY-MM-DD
```

### 7.2 Template → Universal

**触发条件：** 同一规律出现在 3+ 个 Template 中。

**Universal Rules 格式：**
```json
{
  "rules": [
    {
      "id": "universal-001",
      "rule": "具体公司名/产品名作为关键词比泛词命中率更高",
      "based_on": ["template-ai-pm", "template-backend"],
      "confidence": "high",
      "created_at": "2026-04-10"
    }
  ]
}
```

### 7.3 新搜索的策略复用

```
新 JD 进来
  → 按职位关键词匹配最相关 Template → 获取基础策略
  → 应用 Universal 规则 → 过滤已知无效做法
  → 生成 Instance 策略 → 用户协作确认
  → 执行 → 迭代 → 沉淀回 Template 和 Universal
```

### 7.4 用户可控原则

- AI 建议提升（Instance → Template → Universal），用户决定是否采纳
- Template 至少需要 3 次同类搜索才生成，避免以偏概全
- 用户可以手动创建/编辑/删除 Template

## 8. 岗位感知

### 8.1 渠道推荐

| 岗位类型 | 首选渠道 | 辅助渠道 | 排除渠道 |
|---------|---------|---------|---------|
| 技术开发 | GitHub, Google | 技术社区, LinkedIn | Scholar |
| 产品经理 | LinkedIn, Google | 行业媒体, 即刻 | GitHub, Scholar |
| 设计师 | Dribbble, Behance | 小红书, LinkedIn | GitHub, Scholar |
| 管理/高管 | LinkedIn, 行业媒体 | Google | GitHub |
| 学术/研究 | Scholar, 知网 | Google, GitHub | — |

岗位类型通过 JD 的职位关键词自动判断。用户可手动覆盖。

### 8.2 评估维度适配（供 screen Skill 使用）

| 岗位类型 | 维度 1 (30%) | 维度 2 (25%) | 维度 3 (20%) | 维度 4 (15%) | 维度 5 (10%) |
|---------|------------|------------|------------|------------|------------|
| 技术岗 | 岗位匹配度 | 技能覆盖率 | 经验深度 | 行业背景 | 稳定性 |
| 产品岗 | 产品思维 | C端经验 | AI理解力 | 行业洞察 | 执行力 |
| 设计岗 | 设计能力 | 作品集 | 审美 | 行业经验 | 协作力 |

> 注：评估维度适配为 screen Skill 的改动建议，不在本次 public-search 重写范围内，但设计上需预留接口。
