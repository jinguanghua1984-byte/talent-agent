# public-search Skill 完全重写设计规格

> 基于 2026-04-10 全链路测试讨论，将 public-search 从「单次搜索工具」改造为「策略驱动的协作搜索系统」。

**状态：** Draft v2
**日期：** 2026-04-10
**决策：** 完全重写（非渐进改造）、策略数据存储在 talent-agent 项目内
**修订：** v1 → v2（修复 spec review 的 CRITICAL/HIGH 问题）

---

## 1. 问题与目标

### 现状问题

| # | 问题 | 影响 |
|---|------|------|
| P1 | 策略是 AI 内部步骤，用户无法参与 | 搜索方向偏离用户意图 |
| P2 | 无引导入口 | 没有 JD 也没有策略时盲搜 |
| P3 | 一次性执行，无复盘迭代 | 同样的错误反复犯 |
| P4 | 渠道/关键词不区分岗位类型 | 产品经理搜 GitHub 无意义 |
| P5 | 搜索经验不积累 | 每次 from scratch |
| P6 | 不考虑搜索成本 | Token 消耗无感知，浪费真金白银 |

### 设计目标

1. **策略先行**：没有确认的策略不执行搜索
2. **人机协作**：策略是用户和 AI 共同编辑的产物
3. **效果可衡量**：每个策略元素的命中率和 token 成本可归因
4. **经验可复用**：搜索策略三层沉淀（Instance → 策略模板 → Universal）
5. **岗位感知**：根据岗位类型自动调整渠道和评估维度
6. **成本感知**：每次搜索记录 token 消耗，支持成本效益分析

## 2. 触发与入口

### 触发方式

```
/public-search [JD文本 | JD ID | 搜索策略 | 空]
```

### 入口判断

| 用户输入 | 路径 | 后续 |
|---------|------|------|
| JD 文本 / JD ID | JD 驱动 | AI 生成策略 → 用户审查 → 确认 |
| 搜索策略描述（如"用LinkedIn搜AI产品经理"） | 策略驱动 | AI 审查 → 优化建议 → 用户确认 |
| 空 / 岗位描述 | 引导模式 | AI 引导补充信息 → 生成策略 |

### 引导模式输出

```
你想搜什么人？可以给我：
1. 一个 JD（文本或 ID）
2. 你的搜索思路
3. 目标岗位和核心要求，我帮你生成策略
```

### 引导模式退出

用户输入「取消」「算了」或空消息时，终止流程并提示「随时可以再来」。

## 3. 协作策略生成

### 3.1 JD 驱动路径

```
Step 1: 读取 JD → 提取关键信息（职位、技能、行业、年限、城市）
Step 2: 匹配策略模板（如有）→ 获取推荐策略
Step 3: 应用 Universal 规则 → 过滤已知无效做法
Step 4: 生成初版策略文件 → 展示给用户
Step 5: 用户审查、补充、修改 → 合并定稿
Step 6: 用户确认 → 进入执行
```

### 3.2 策略驱动路径

```
Step 1: 解析用户搜索策略 → 结构化展示
Step 2: AI 审查 → 给出优化建议（标注"建议"非"决定"）
Step 3: 用户采纳/拒绝 → 定稿
Step 4: 用户确认 → 进入执行
```

### 3.3 协作原则

- 策略是可编辑的协作产物，不是 AI 单方面输出
- 用户始终有最终决定权
- AI 的角色是审查、补充、优化，不是替代

### 3.4 策略文件格式

策略保存为「活文档」，搜索过程中持续更新：

```markdown
# 搜索策略：{JD标题}

## 元信息
- JD ID: {id}
- 创建时间: {timestamp}
- 当前轮次: {N}
- 来源模板: {template-id 或 "无"}
- 最后活跃时间: {timestamp}

## 目标画像
（1-3 种候选人画像，每种包含核心特征）

## 搜索策略
### 关键词矩阵
| # | 关键词 | 渠道 | 优先级 | 状态 | 备注 |
|---|--------|------|--------|------|------|
| 1 | "AI Agent" "产品经理" | Google | 高 | active | |
| 2 | "百万DAU" "产品经理" | Google | 高 | removed | 轮次2: 命中率0% |

### 渠道计划
| 渠道 | 搜索条数 | 权重 |
|------|---------|------|
| LinkedIn | 3 | 高 |

## 执行记录
### 轮次 1 ({timestamp})
#### 归因表
| 策略元素 | 搜索量 | 候选人 | 噪音 | 命中率 | Token消耗 | 成本/候选人 | 状态 |
|---------|--------|--------|------|--------|----------|------------|------|

#### 覆盖缺口
| JD 要求 | 已匹配 | 缺口 |
|---------|--------|------|

#### AI 反思
（本轮学到了什么、成本分析、优化建议）

### 累计归因
| 策略元素 | 累计搜索量 | 累计候选人 | 累计命中率 | 累计Token | 平均成本/人 | 状态 |
|---------|-----------|-----------|-----------|----------|------------|------|

### 轮次 2 ({timestamp})
...
```

**关键字段说明：**
- `状态` 枚举值：`active` / `removed` / `modified`，备注列记录原因
- `最后活跃时间`：每次更新策略时同步更新，用于跨会话恢复判断
- `累计归因`：多轮搜索后汇总每个策略元素的累计效果

### 3.5 存储路径

```
data/search-strategies/
├── instances/              # 单次搜索策略（活文档）
│   └── {date}-{slug}.md
├── templates/              # 搜索策略模板（N次后提炼）
│   └── {template-id}.md
└── universal-rules.json    # 通用规则（跨模板共性）
```

### 3.6 跨会话恢复

策略文件是跨会话的恢复点：

1. **新会话启动**：用户给出 JD ID 或策略文件路径，AI 读取策略文件
2. **恢复判断**：检查「最后活跃时间」，如果距当前超过 24 小时，提示「这是一次 X 天前的搜索，是否继续？」
3. **并发控制**：策略文件中记录「当前轮次」，新会话开始时检查轮次号。如果轮次号与文件不一致（说明有其他会话在更新），提示冲突，建议用户选择：
   - 基于当前文件继续
   - 创建新策略文件
   > 注意：并发控制为 best-effort 级别（非 ACID），适用于单人使用场景。
4. **退出机制**：用户输入「取消」「算了」或空消息时终止流程

## 4. 执行搜索

### 4.1 执行逻辑

按关键词矩阵逐条执行，每条独立记录：

```
1. 调用搜索工具（WebSearch / GitHub / jina-reader）
2. 提取页面内容，识别候选人信息
3. 记录：搜索量、候选人数、噪音数
4. 候选人去重，合并 sources
5. 通过 Token Tracker 获取本次操作的 token 消耗
```

### 4.2 Token 消耗追踪

**方案：基于 Claude Code OpenTelemetry 遥测 + 轻量级 Python 接收器**

Claude Code 原生支持 OpenTelemetry 导出，每次 API 请求自动上报 `input_tokens` / `output_tokens` / `cost_usd`。

**环境配置（一次性）：**

```bash
export CLAUDE_CODE_ENABLE_TELEMETRY=1
export OTEL_LOGS_EXPORTER=otlp
export OTEL_EXPORTER_OTLP_PROTOCOL=http/json
export OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4318
```

**本地接收器：** `scripts/token-tracker.py`，纯 Python stdlib 实现的 OTLP HTTP 接收器，零外部依赖。接收 Claude Code 的 `api_request` 事件，提取 token 数据写入 `data/token-tracker/tokens.jsonl`。

启动命令：`python scripts/token-tracker.py`（可选 `--port 4318 --output data/token-tracker/tokens.jsonl`）

**Token Tracker 接口：** Skill 通过读取 `data/token-tracker/tokens.jsonl` 获取 token 数据，每行一个 JSON 对象：

```json
{"timestamp": "1712736000000000000", "prompt_id": "uuid", "input_tokens": 8432, "output_tokens": 1203, "cache_read_tokens": 5000, "cache_creation_tokens": 2000, "cost_usd": 0.0523, "model": "claude-sonnet-4-6"}
```

**归因方式：** 每个 AI turn 对应一个 `prompt_id`，Skill 在执行搜索前后记录当前 turn 的 prompt_id，从 tokens.jsonl 中按 prompt_id 聚合 token 消耗，归因到对应的策略元素。prompt_id 获取方式待实现时验证，如不可用则退化为按时间窗口归因。

**降级方案：** 如果 token-tracker.py 未运行或 tokens.jsonl 不存在，归因表中 Token 列显示「未配置」，成本分析跳过。不使用代理指标。

**OTEL 环境变量说明：** 以上环境变量名称以 Claude Code 实际文档为准。`http/json` 协议使用 HTTP POST（端口 4318），无需 gRPC 依赖。

### 4.3 候选人写入

用户确认搜索结果后，写入 `data/candidates/`：

1. 为每个候选人创建临时 JSON 文件
2. 调用 `data-manager.py candidate create` 写入
3. 清理临时文件
4. 多渠道发现的同一候选人合并 sources 而非重复创建
5. 去重逻辑复用 `data-manager.py candidate dedup` 现有命令

## 5. 搜索反馈

### 5.1 层 1：归因表

每个策略元素（关键词+渠道组合）的效果和成本可衡量：

```
| 策略元素 | 搜索量 | 候选人 | 噪音 | 命中率 | Token消耗 | 成本/候选人 | 状态 |
|---------|--------|--------|------|--------|----------|------------|------|
| "豆包" PM LinkedIn | 8 | 2 | 6 | 25% | 3.2K | 1.6K | ✅ 保留 |
| "AI PM" Google | 23 | 3 | 20 | 13% | 15.1K | 5.0K | ⚠️ 成本偏高 |
| "百万DAU" Google | 15 | 0 | 15 | 0% | 12.3K | ∞ | ❌ 建议移除 |
```

Token 消耗为 0 时显示「未配置」。

### 5.2 层 2：覆盖缺口

对照 JD 要求，分析匹配情况：

```
| JD 要求 | 已匹配 | 缺口 |
|---------|--------|------|
| 5-10年产品经验 | ✅ 5人 | — |
| C端APP独立负责 | ⚠️ 2人 | 不确定是否独立负责 |
| 百万DAU+ | ❌ 0人 | 公开渠道难以验证 |
```

### 5.3 层 3：AI 反思 + 成本分析 + 优化建议

AI 从数据中自动分析，给建议但不直接改策略：

```

• 关键词：带公司名(豆包)比泛词(AI产品经理)更精准

💰 成本分析

• 最高效："豆包" LinkedIn（1.6K/人）
• 最浪费："百万DAU" Google（12.3K，0产出）

💡 优化建议
A. 关键词：去掉"百万DAU"，加"ChatGPT Notion AI 产品经理"
B. 渠道：提高 LinkedIn 优先级

你的想法？
```

## 6. 迭代循环

### 用户选择

| 用户反馈 | 动作 |
|---------|------|
| 采纳某些建议 | 更新策略文件 → 再搜一轮 |
| 提出自己的优化 | AI 审查 → 更新策略 → 再搜一轮 |
| 满意当前结果 | 结束搜索 → 进入 screen/report |
| 放弃 | 在策略文件中记录放弃原因和轮次，不删除文件 |

### 每轮迭代操作

1. 更新策略文件中的「执行记录」（追加新一轮）
2. 更新关键词矩阵的「状态」和「备注」列
3. 重新计算「累计归因」表
4. 重新分析覆盖缺口
5. 更新「最后活跃时间」
6. Git commit 策略文件变更

### 放弃记录

放弃时在策略文件末尾追加：

```markdown
## 搜索结果：放弃
- 放弃轮次：{N}
- 放弃原因：{用户描述}
- 沉淀经验：{AI 从本轮搜索中学到的通用经验}
```

放弃的 Instance 不参与策略模板的统计数据计算，但其「沉淀经验」部分可以被 AI 提取用于 Universal 规则。

## 7. 策略沉淀

### 7.1 核心概念

**「同类搜索」的定义：** 两次搜索的 JD 全文相似度超过阈值（初始阈值：0.6，可调）。相似度计算方式：提取 JD 的职位名称、技能要求、行业、经验要求等字段，计算加权相似度。

**策略模板不是「岗位模板」：** 模板捕获的是近似搜索策略的共性特征，不局限于同一岗位。两个不同岗位如果搜索策略高度相似，可以归入同一模板。

### 7.2 Instance → 策略模板

**触发条件：** 当 N 个 Instance 的 JD 两两相似度均超过阈值（初始 N=3，可调），触发模板创建提议。已有模板下积累 N+ 次新 Instance 时，建议更新模板统计数据。

**AI 提议：**
```
💡 发现 3 次近似搜索的共性：
• LinkedIn 命中率稳定在 20%+
• GitHub 命中率接近 0%
• 带具体产品名的关键词效果最好

共同特征：C端产品经理、AI行业、上海

是否创建搜索策略模板？
```

**策略模板文件格式：**
```markdown
# 搜索策略模板：{template-id}

## 共同特征
（从相关 Instance 的 JD 中提取的共性，用于新 JD 匹配判断）
- 职位类型：产品经理
- 行业：AI
- 关键词：C端、移动端、Agent
- ...

## 推荐渠道
1. LinkedIn（权重：高，命中率 22%）
2. Google（权重：高，命中率 13%）

## 推荐关键词模式
- 具体产品名 > 泛词
- 带 AI 公司名 > 不带

## 排除渠道
- GitHub（命中率 < 2%）

## 排除关键词模式
- 不可量化指标（百万DAU等）命中率 0%

## 统计数据
- 关联 Instance 数：3
- 平均命中率：18%
- 平均成本/候选人：4.2K tokens
- 最后更新：YYYY-MM-DD
```

### 7.3 新 JD 匹配策略模板

匹配逻辑与「同类判定」是同一套：

```
1. 提取新 JD 的特征（职位、技能、行业、经验）
2. 与每个策略模板的「共同特征」计算相似度
3. 相似度超过阈值（0.6）的模板为候选
4. 多个候选模板时，按相似度排序，展示给用户选择
5. 用户也可选择「不使用模板」从零生成
```

### 7.4 策略模板 → Universal 规则

**触发条件：** 同一规律出现在 3+ 个策略模板中。

**Universal Rules 格式：**
```json
{
  "rules": [
    {
      "id": "universal-001",
      "rule": "具体公司名/产品名作为关键词比泛词命中率更高",
      "based_on": ["template-001", "template-003"],
      "confidence": "high",
      "created_at": "2026-04-10",
      "status": "active"
    }
  ]
}
```

**规则生命周期：**
- `status` 枚举：`active` / `deprecated`
- AI 定期检查 Universal 规则是否与最新 Instance 数据一致
- 连续 5 次 Instance 违反某规则 → AI 建议标记为 `deprecated`

### 7.5 沉淀全景

```
Instance（单次策略活文档）
  ↓ N 次相似 Instance（JD 全文相似度 > 0.6）
策略模板（共同特征 + 推荐策略 + 统计）
  ↓ 同一规律出现在 3+ 个模板
Universal 规则（跨模板共性）
```

**反向应用：**
```
新 JD → 匹配策略模板 → 获取推荐策略 → 应用 Universal 规则 → 生成 Instance → 用户协作 → 执行 → 迭代 → 沉淀
```

### 7.6 用户可控原则

- AI 建议提升（Instance → 策略模板 → Universal），用户决定是否采纳
- 用户可以手动创建/编辑/合并/删除策略模板
- 用户可以手动添加/删除/废弃 Universal 规则
- 相似度阈值用户可调

## 8. 岗位感知

### 8.1 渠道推荐

| 岗位类型 | 首选渠道 | 辅助渠道 | 排除渠道 |
|---------|---------|---------|---------|
| 技术开发 | GitHub, Google | 技术社区, LinkedIn | Scholar |
| 产品经理 | LinkedIn, Google | 行业媒体, 即刻 | GitHub, Scholar |
| 设计师 | Dribbble, Behance | 小红书, LinkedIn | GitHub, Scholar |
| 管理/高管 | LinkedIn, 行业媒体 | Google | GitHub |
| 学术/研究 | Scholar, 知网 | Google, GitHub | — |

**岗位类型判断：** 从 JD 职位名称中提取关键词，匹配上表。模糊情况（如「全栈工程师」）默认匹配最接近的类型，用户可手动覆盖。覆盖记录在策略文件的元信息中。

### 8.2 评估维度适配（供 screen Skill 使用）

| 岗位类型 | 维度 1 (30%) | 维度 2 (25%) | 维度 3 (20%) | 维度 4 (15%) | 维度 5 (10%) |
|---------|------------|------------|------------|------------|------------|
| 技术岗 | 岗位匹配度 | 技能覆盖率 | 经验深度 | 行业背景 | 稳定性 |
| 产品岗 | 产品思维 | C端经验 | AI理解力 | 行业洞察 | 执行力 |
| 设计岗 | 设计能力 | 作品集 | 审美 | 行业经验 | 协作力 |

> 注：评估维度适配为 screen Skill 的改动建议，不在本次 public-search 重写范围内，但设计上需预留接口（策略模板中可选输出 `evaluation_dimensions` 字段）。
