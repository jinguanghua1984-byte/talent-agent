# 一人猎头公司的 AI 化端到端商业解决方案

> 日期：2026-05-21
> 目标：基于 `docs/research`、本仓库产品定位和最近开发实现，设计一个由 1 名负责人经营、AI 深度参与的端到端猎头商业系统。
> 约束：方案追求高自动化，但不绕过招聘平台登录、验证码、安全页、风控和平台条款；客户关系、候选人信任、最终判断和关键写库/外发仍由人负责。

## 1. 核心判断

“一人猎头公司”不是把猎头全流程无人化，而是把传统猎头公司的 researcher、sourcer、数据助理、报告助理、运营助理和部分项目经理职能压缩进 AI 系统，让 1 名负责人把精力集中在客户信任、岗位判断、候选人沟通和成交上。

本项目已经接近这个方向：当前定位是“猎头顾问的综合 AI 助理”，覆盖公域搜索、平台匹配、筛选评估和推荐报告；仓库已经有运行时中立 workflow、人才库、脉脉 campaign、浏览器扩展、评分/精排、Feishu 交付和同步包能力。近期 LLM 推理岗位实践已经证明：从岗位深挖、脉脉寻访、详情补全、精排、外联队列、主库同步到飞书交付可以跑通，只需要把它产品化为稳定服务包。

商业上最合理的切入点不是做通用低价招聘外包，而是做高复杂度、高人才稀缺度、高解释成本的技术岗位，例如 AI Infra、大模型推理、训练框架、GPU 平台、算法平台、基础软件和 CTO/技术负责人级岗位。AI 在这些岗位上能显著放大“行业研究 + 搜索策略 + 证据化评估”的效率，人负责高信任沟通和成交。

## 2. 商业定位

### 2.1 目标客户

第一阶段只服务 3 类客户：

| 客户类型 | 典型需求 | 为什么适合一人 AI 猎头 |
| --- | --- | --- |
| AI 创业公司 | 关键技术岗位，候选人少，JD 模糊，需要快速 mapping | 需要行业理解、快速搜索和高质量首批名单，不需要大型 RPO |
| 大厂创新业务/云厂商/AI 平台团队 | 细分技术方向补强，要求候选人证据充分 | AI 可以把岗位模型、公司池、关键词包和候选人证据结构化 |
| 早期 CEO/CTO | 不知道市场上谁合适，需要先校准画像 | 岗位深挖和人才地图可先作为付费咨询产品 |

暂不优先做：

- 高量低价岗位，如销售、客服、基础运营。
- 完全标准化校园招聘。
- 客户没有明确招聘预算、只想“看看市场”的需求，除非转成付费 mapping。

### 2.2 服务产品

| 产品 | 交付物 | 收费建议 | 适用场景 |
| --- | --- | --- | --- |
| 岗位深挖 Sprint | 岗位本质、能力模型、公司池、关键词包、面试问题、薪酬/吸引点假设 | 固定项目费 5,000-20,000 元 | 客户 JD 模糊或首次招聘该岗位 |
| 人才地图 Mapping | 目标公司池、长名单、候选人分层、Top 人选证据表 | 固定项目费 10,000-50,000 元 | 先判断市场供给，不承诺入职 |
| 端到端寻访 | 岗位深挖、搜索、详情补全、精排、外联、推荐、跟进 | Retainer + 成功费，或成功费 15%-25% 年薪 | 高价值岗位 |
| 人才池运营 | 周期性更新人才库、候选人 nurture、机会提醒 | 月费 5,000-30,000 元 | 长期招聘同类岗位 |

第一年建议以“项目制 + 成功费”为主，软件作为服务交付能力，不先卖 SaaS。原因是当前差异化来自方法论、数据资产和交付速度，而不是多人协同系统。

### 2.3 单位经济模型

以下是经营假设，不依赖外部市场规模数据：

| 指标 | 传统手工模式 | AI 化目标 |
| --- | ---: | ---: |
| 岗位 intake 到第一版人才地图 | 2-5 天 | 0.5-1 天 |
| 每个岗位人工研究/搜索/整理 | 60-120 小时 | 20-45 小时 |
| 首批可推荐人选产出 | 7-14 天 | 2-5 天 |
| 单人同时活跃岗位 | 2-4 个 | 5-8 个 |
| 候选人证据整理 | 手工摘要，质量不稳定 | 自动结构化，人工审校 |

目标不是把人工时间降到 0，而是把低价值整理时间减少 60% 以上，让负责人把时间投向：

- 客户需求校准。
- 候选人初聊和信任建立。
- 推荐前事实核验。
- 面试/offer 推进。
- 客户反馈转化为规则。

## 3. 端到端业务闭环

### 3.1 总流程

```text
获客/定位
  -> 客户需求 intake
  -> 岗位深挖和人才画像
  -> 搜索策略和关键词包
  -> 多渠道寻访/人才库召回
  -> 列表去重和粗筛
  -> 详情补全和证据抽取
  -> 精排和外联优先级
  -> 人工确认外联策略
  -> 候选人沟通和意愿确认
  -> 客户推荐报告
  -> 面试/offer 跟进
  -> 数据回流和规则进化
```

### 3.2 每一环的人机分工

| 环节 | 人负责 | AI 负责 | 当前仓库基础 |
| --- | --- | --- | --- |
| 获客与定位 | 选择客户、定价、承诺边界、签约 | 生成行业观点、案例报告、客户邮件草稿 | `docs/research` 可转化为内容营销资产 |
| 需求 intake | 访谈客户、追问业务优先级、确认硬门槛 | 从 JD/访谈纪要抽取目标角色、能力模型、关键词、排除项 | `docs/design-discussions/2026-05-21-llm-inference-role-deep-dive.md`、`skills/maimai-talent-search-campaign/SKILL.md` |
| 搜索策略 | 批准搜索计划、决定平台和风险边界 | 生成公司池、岗位别名、关键词包、wave 计划、run-policy | `maimai-talent-search-campaign` 合同文件 |
| 平台寻访 | 登录、处理验证码、安全页、人工确认计划 | 在安全边界内执行 search wave、raw 落盘、断点恢复 | `agents/workflows/maimai-unattended-campaign/AGENT.md`、`scripts/maimai_campaign_orchestrator.py` |
| 人才库召回 | 判断历史人选是否可复用、避免误合并 | 本地检索、只读匹配、评分、生成 TopN | `agents/workflows/talent-library/AGENT.md`、`scripts/llm_inference_talent_match.py` |
| 粗筛/精排 | 调整权重、确认最终推荐逻辑 | 五维评分、A/B/C/淘汰分层、证据摘要、风险标记 | `agents/workflows/screen/AGENT.md`、`scripts/maimai_ai_infra_rank.py` |
| 详情补全 | 决定是否抓详情、处理平台阻断 | 详情目标包、健康检查、详情抓取、dry-run/apply 到 campaign DB | `extensions/maimai-scraper`、`scripts/maimai_ai_infra_detail_live_gate.py` |
| 外联 | 决定是否联系、联系人设定、真实消息发送 | 生成外联角度、优先级队列、跟进提醒 | `scripts/maimai_ai_infra_outreach_export.py`、Feishu delivery |
| 候选人沟通 | 真实沟通、判断动机、薪资、风险、背调 | 生成问题清单、通话纪要、候选人材料包 | `wechat-chat-sync` workflow 可作为后续聊天记录索引 |
| 客户推荐 | 最终选择推荐人选、解释风险 | 推荐报告、横向对比、证据链、飞书文档/表格 | `agents/workflows/report/AGENT.md`、`scripts/feishu_delivery_package.py` |
| 复盘进化 | 判断客户反馈是否合理、沉淀规则 | 提取客户偏好、更新评分规则、复用到后续岗位 | `screen` 的 corrections/learned rules 思路 |

## 4. 技术架构

### 4.1 原则

1. 运行时中立：业务 workflow 放在 `agents/workflows/`，具体 agent 只做 adapter。
2. Campaign 与主库隔离：长任务先写 `data/campaigns/<campaign_id>/talent.db`，主库写入必须单独 dry-run、备份和确认。
3. 可恢复：搜索和详情都以 raw 文件、ledger 和 continuation plan 为事实来源。
4. 可审计：每次搜索、导入、评分、详情、交付都留 JSON/Markdown/CSV。
5. 人审门禁：平台安全、主库写入、候选人真实外联、客户最终推荐由人承担责任。

### 4.2 目标架构

```text
业务入口层
  - 岗位深挖 / campaign skill / talent-library / report

编排层
  - maimai-unattended-campaign workflow
  - maimai_campaign_orchestrator
  - run-policy / events / continuation-plan

采集层
  - maimai-scraper 扩展
  - Boss 被动捕获规划
  - 公域采集后续可接 Crawl4AI/Crawlee/ScrapeGraphAI

数据层
  - campaign DB
  - 主人才库 data/talent.db
  - source_profiles / candidate_details / match_scores / sync bundle

智能层
  - JD 深挖
  - 搜索策略
  - 规则评分 + LLM 辅助解释
  - 人才库召回
  - 外联角度和推荐报告

交付层
  - 本地 Markdown/JSON/CSV
  - Feishu Doc/Sheet/IM
  - 后续可接 n8n/Cal.com/Chatwoot
```

### 4.3 现有能力映射

| 能力 | 当前状态 | 商业方案中的作用 |
| --- | --- | --- |
| 产品定位 | README 明确面向独立猎头顾问和小型猎头公司 | 一人公司正是核心用户画像 |
| 人才库 | `talent-library`、`TalentDB`、导入、详情、微信同步、sync bundle | 长期复利资产 |
| 脉脉寻访 | skill + unattended workflow + orchestrator | 高价值岗位的主寻访引擎 |
| 浏览器扩展 | `maimai-scraper` 列表、分页、详情、导出、日志 | 平台数据采集和详情补全 |
| Boss 渠道 | 已有设计，第一版只做列表闭环和被动详情 | 第二渠道，避免单平台依赖 |
| 评分与精排 | `screen`、`maimai_ai_infra_rank`、LLM 推理只读匹配脚本 | 从长名单到推荐名单 |
| 交付包 | `feishu_delivery_package.py`、Feishu 文档/表格 | 客户可读交付和跟进队列 |
| 最近验证 | LLM 推理 campaign 完成 86/86 详情，最终强推荐 6、推荐 13、观察 27；主库已有 5,497 候选人 | 证明高复杂技术岗闭环已可跑通 |

### 4.4 开源组件采用策略

根据 `docs/research`，开源 HR/招聘项目的成熟度分布不均：简历工具和通用 AI workflow 很成熟，真正面向猎头的自动寻访、触达和 ATS 很薄弱。因此策略是“吸收能力，不替换核心”。

| 组件/方向 | 建议 | 用法 |
| --- | --- | --- |
| Docling / Unstructured | 可引入 | 简历、附件、面试记录结构化 |
| Resume Matcher | 可引入或借鉴 | JD-简历匹配解释、差距分析 |
| Haystack | 中期引入 | 人才库向量召回、岗位知识库、rerank |
| Dify / Langflow | 可作为业务入口原型 | 岗位 intake、内部问答、流程可视化 |
| n8n | 可作为自动化胶水 | 状态变更、提醒、日历、消息编排 |
| Cal.com | 可引入 | 候选人初聊/面试排期 |
| Twenty / NocoDB | 可参考，不急于替换 | 若需要可视化 CRM/表格后台，可接主库只读视图 |
| OpenCATS / Frappe HRMS | 只参考 | 完整 ATS/HRMS 对一人公司过重 |
| LinkedIn 自动寻访 / AIHawk 类工具 | 暂不直接使用 | 只研究 fit score、消息生成和流程设计，不直接跑平台自动化 |
| Firecrawl | 谨慎 | 能力强但 AGPL 影响闭源商业集成，可独立评估 |

## 5. 人负责什么，AI 负责什么

### 5.1 人必须负责

1. 客户选择和商业承诺：是否接单、交付范围、收费方式、成功定义。
2. 岗位真实优先级：哪些要求是硬门槛，哪些是可妥协。
3. 平台合规操作：登录、验证码、安全页、风控处理、平台条款判断。
4. 候选人信任：首次真实触达、动机沟通、薪资、家庭/地域约束、隐私授权。
5. 事实核验：推荐前确认候选人履历、意愿和关键风险。
6. 客户说服：解释候选人亮点、风险和替代方案。
7. Offer 推进：薪资谈判、反 offer、入职风险。
8. 最终数据写入边界：主库写入、删除、合并、同步。

### 5.2 AI 可以负责

1. 读 JD 和访谈纪要，抽取岗位画像。
2. 生成公司池、关键词包、职位别名和排除项。
3. 生成 campaign 合同、run-policy、search wave 和 detail pack。
4. 执行可恢复的列表采集、详情补全、dry-run/apply。
5. 人才库召回、去重、分层和证据摘要。
6. 生成候选人推荐理由、风险标记、面试问题。
7. 生成外联草稿和个性化切入点。
8. 生成客户报告、Feishu 文档、候选人 Sheet、外联队列。
9. 跟踪任务状态、提醒下一步、沉淀客户反馈。

### 5.3 不应该交给 AI 独立负责

1. 自动绕过验证码、登录、安全页、429/403/432。
2. 未经人工授权批量发送候选人消息。
3. 自动承诺候选人或客户条件。
4. 自动判定敏感个人信息是否可对外传播。
5. 自动删除候选人或强合并跨平台身份。
6. 把 AI 评分当成最终录用/淘汰结论。

## 6. 商业运营打法

### 6.1 获客策略

最适合一人公司的获客方式是“专业内容 + 小型付费诊断 + 高质量交付样板”。

1. 每周发布 1 篇岗位/人才地图洞察，例如“大模型推理岗位到底找什么人”。
2. 给潜在客户提供低价或免费的 30 分钟 intake，但岗位深挖报告收费。
3. 用匿名化样板展示交付能力：岗位画像、候选人分层、证据表、外联队列，而不是展示候选人隐私。
4. 把成功案例转成“同类岗位可复用搜索资产”，逐步形成垂直赛道壁垒。

### 6.2 交付 SOP

每个端到端岗位建议采用 7 天首轮节奏：

| 天数 | 目标 | 主要动作 |
| --- | --- | --- |
| D0 | 签约/需求确认 | 客户访谈、JD 深挖、硬门槛确认 |
| D1 | 搜索计划 | 公司池、关键词包、run-policy、搜索计划确认 |
| D2-D3 | 长名单 | 平台搜索、人才库召回、列表粗筛 |
| D3-D4 | 详情和精排 | 详情补全、A/B/C/淘汰、外联优先级 |
| D4-D6 | 外联和初聊 | 人审后触达、候选人意愿和关键事实核验 |
| D5-D7 | 首批推荐 | 客户报告、对比表、面试问题、风险说明 |
| 后续 | 跟进成交 | 面试反馈、规则修正、补充寻访 |

### 6.3 核心 KPI

| 类型 | KPI | 目标 |
| --- | --- | --- |
| 速度 | intake 到搜索计划 | < 24 小时 |
| 速度 | intake 到首批可推荐 | 2-5 天 |
| 供给 | 每个岗位长名单 | 100-500 人，视岗位窄度 |
| 质量 | A/B 档详情完成率 | > 95%，平台阻断除外 |
| 质量 | 首批推荐面试通过率 | > 30% |
| 触达 | P0/P1 回复率 | 按渠道和岗位复盘 |
| 成交 | 推荐到 offer 转化 | 按客户/岗位沉淀 |
| 数据 | 候选人 source/profile 完整率 | > 90% |
| 安全 | 非授权主库写入 | 0 |
| 安全 | 平台安全事件继续推进 | 0 |

## 7. 落地路线

### Phase 0：以服务交付验证商业闭环，0-2 周

目标：不做新前端，先用现有仓库稳定交付 3 个真实岗位。

任务：

- 固化岗位深挖模板，覆盖岗位本质、能力模型、公司池、关键词包、面试问题。
- 固化 campaign 交付目录模板和命名规则。
- 每次 campaign 结束自动生成匿名化交付摘要。
- 把 Feishu 候选人 Sheet 和外联队列表头固定成客户可读版本。
- 建立“客户反馈 -> 评分规则调整 -> 下次复用”的记录格式。

验收：

- 至少 3 个岗位能在 7 天内交付首批推荐。
- 每个岗位都有完整本地 evidence：requirements、strategy、run-policy、raw、rank、report、outreach queue。
- 无未经授权主库写入、无敏感 raw 上传到 Feishu。

### Phase 1：一人工作台，2-6 周

目标：把命令行和散落文档收敛成负责人每天可用的操作台。

任务：

- 做一个本地 Dashboard：岗位状态、campaign 阶段、阻断原因、今日待办、候选人队列。
- 增加“一键生成客户交付包”和“一键生成外联审批队列”。
- 增加候选人状态字段：待外联、已外联、已回复、已初聊、已推荐、面试中、offer、关闭。
- 对接 Cal.com 或日历，用于候选人初聊排期。
- 对接 n8n 做提醒和状态回写，但真实外发仍需人审。

验收：

- 负责人不需要记底层脚本参数。
- 任一候选人可以追溯到来源、详情、评分证据、外联记录和客户反馈。
- 每天的人工待办不超过一个可执行队列。

### Phase 2：智能人才库和知识库，1-3 个月

目标：让数据资产产生复利，而不是每个岗位从零开始。

任务：

- 引入简历/附件结构化解析，可评估 Docling/Unstructured。
- 建立岗位画像库、公司池库、关键词包库和成功候选人样本库。
- 引入向量召回或检索增强，可评估 Haystack。
- 建立跨岗位相似候选人推荐。
- 将客户反馈转成客户级偏好规则。

验收：

- 新岗位 intake 后，系统能自动推荐相似历史岗位、相似候选人和已有公司池。
- 人才库召回结果能解释“为什么推荐”和“缺什么信息”。
- 客户反馈能影响下次评分，而不是只停留在聊天记录。

### Phase 3：多渠道和商业化包装，3-6 个月

目标：降低单平台依赖，把服务能力包装为可复制产品。

任务：

- 推进 Boss 渠道第一版：列表捕获、导出、dry-run、确认入库；详情只做被动捕获。
- 公域数据采集分层接入 Crawl4AI/Crawlee/ScrapeGraphAI。
- 把岗位深挖 Sprint 和人才地图 Mapping 做成标准报价产品。
- 建立匿名化案例库和行业报告输出节奏。
- 对重点客户提供月度人才池运营。

验收：

- 至少两个稳定渠道进入同一人才库。
- 可按公司、方向、岗位、技术关键词复用历史 mapping。
- 单人月度可稳定运营 5-8 个活跃岗位或 2-3 个深度客户。

## 8. 风险和控制

| 风险 | 表现 | 控制措施 |
| --- | --- | --- |
| 平台合规风险 | 自动化触发验证码、封禁、条款风险 | 不绕过登录/验证码/安全页；真实平台阶段遇 403/429/432/非 JSON 立即停机 |
| 数据隐私风险 | 候选人信息外泄、raw 上传 | Feishu 只上传筛选后交付表，不上传 DB/raw capture；敏感字段最小化 |
| AI 误判风险 | 评分幻觉、候选人被错误淘汰 | AI 评分只做建议；A/B/C 需可解释证据；关键推荐前人工复核 |
| 触达骚扰风险 | 批量消息伤害候选人信任 | 人审外联；限制频率；保留退订/不再联系标记 |
| 单人瓶颈 | 多岗位并发导致沟通质量下降 | AI 只扩后台能力；限制活跃岗位；外联和客户沟通设 SLA |
| 数据污染 | 跨平台误合并、低质量数据覆盖高质量数据 | 同平台 ID 合并，跨平台保守合并；fill-only；主库写入 dry-run/backup |
| 商业定位失焦 | 接太多低价值岗位 | 只接高复杂度、高单价、可复用赛道 |

## 9. 最小可执行版本

如果明天开始经营，最小版本只需要 6 个资产：

1. 一页客户服务说明：我只做高复杂技术岗位，首批交付 7 天内给出。
2. 岗位 intake 模板：岗位目标、硬门槛、目标公司、排除项、薪酬、卖点。
3. 岗位深挖报告模板：岗位本质、能力模型、候选人类型、面试问题。
4. Campaign SOP：`requirements.json`、`strategy.json`、`run-policy.json`、搜索计划、详情计划。
5. 客户交付模板：候选人对比表、推荐理由、风险、外联状态。
6. 复盘模板：客户反馈、候选人反馈、评分规则更新、下次搜索建议。

这个最小版本已经大部分能由当前仓库支撑。下一步不是大改架构，而是把每次成功交付沉淀成固定模板、固定命令、固定审计和固定商业话术。

## 10. 结论

一人猎头公司的可行路径是：用 AI 做“研究、搜索、结构化、排序、报告、提醒”，用人做“信任、判断、外联、谈判、成交”。本仓库的最佳商业化方向是先成为一个服务型 AI 猎头操作系统，而不是先做通用 ATS 或招聘 SaaS。

短期应围绕 AI Infra/大模型技术岗位形成样板：岗位深挖 -> 脉脉/人才库寻访 -> 详情补全 -> 证据化精排 -> 外联队列 -> 飞书交付 -> 客户反馈回流。等 3-5 个岗位跑出稳定转化后，再把 NocoDB/Twenty/n8n/Haystack/Docling 等外部组件按瓶颈逐步接入。

真正的护城河不是“能不能自动抓人”，而是持续积累的岗位知识、人才关系、候选人证据、客户偏好和可审计交付流程。
